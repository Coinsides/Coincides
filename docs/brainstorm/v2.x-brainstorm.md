# Coincides v2.x Brainstorm

> Status: Brainstorm draft / 草稿池
> Purpose: Record early product and architecture thinking for the v2.x NoteBlock Library, Course Material Library, material reconciliation, note organization, review projections, concept-focused learning material views, and future migration from the v1.x Card/Deck model.
> Scope note: This document is not the formal roadmap. Mature ideas can later be promoted into `Coincides-Roadmap.md`, `docs/v2-note-system-draft.md`, release plans, or ADRs.

---

## 1. Working Thesis

Coincides v2.x should move from a card-centered learning system toward a course-rooted learning material system.

The current v1.x Card/Deck system should be understood as an early prototype of structured learning blocks, not as a permanent peer of the future Note system. It validated typed learning units, KaTeX rendering, FSRS review, source-linked AI generation, and Proposal -> Review -> Apply.

The v2 target should become:

```text
Workspace / Local Profile
  -> Course
    -> Source Library
    -> Course Material Library
    -> NoteBlock Library
    -> Projection System
```

Course is the learning-domain root. Workspace stores global/user context. Agent Memory stores user preferences. Course Material Library stores learning knowledge.

---

## 2. Generational Model

### 2.1 v1.x Card/Deck: Early Prototype

The current Card/Deck system is the ancestor of the future NoteBlock system.

Cards currently hold limited structured academic content:

- definitions;
- theorems;
- formulas;
- examples;
- general notes.

Decks and sections provide a first attempt at grouping these objects. This is useful, but it is not yet a real note system:

- there is no complete title/paragraph/diagram/answer/proof structure;
- there is no natural reading order across all blocks;
- there are no transitions or explanations between blocks;
- examples, answers, diagrams, and formulas are not fully independent fragments;
- a deck is a collection, not a reconstructed learning narrative;
- a card is optimized around review, not around preserving the professor's or student's note structure.

The v1.x system should be treated as a learning experiment and migration source, not a structure that must be protected at all costs.

### 2.2 v2.x NoteBlock Library: Core Substrate

The NoteBlock Library should become the canonical substrate for reusable learning fragments.

Candidate block types include:

- title;
- heading;
- paragraph;
- definition;
- theorem;
- proof;
- proof step;
- formula;
- example;
- answer;
- exercise;
- question;
- diagram crop;
- image crop;
- source excerpt;
- sidenote;
- annotation;
- divider;
- summary.

Every block should be source-aware and editable. It should preserve enough context to be reused later in organized notes, review sets, exercise sets, concept views, and future knowledge maps.

### 2.3 v2.x New Card/Review System: Projection Layer

The future Card/Review System should be an upper-layer projection over NoteBlock Library, not a permanent separate source of truth.

Examples of review projections:

- Review Card Set;
- Knowledge Checklist;
- Exercise Set;
- Example Set;
- Formula Sheet;
- Theorem-Proof List;
- Proof Practice Set;
- Exam Review Set.

A block can appear in multiple projections. AI may propose review projections, but the user should approve them before they become official.

## 3. Course Material Library

Course Material Library should become a first-class asset under each Course, alongside schedules, goals, tasks, notes, review sets, and concepts.

It exists because real learning materials are messy:

- users upload materials in multiple batches;
- upload order may not match learning order;
- textbooks, notes, slides, scans, and problem sets may overlap heavily;
- users may only need selected chapters, weeks, pages, or segments;
- some material may be missing, duplicated, out of scope, or intentionally excluded.

The library should let the user and AI answer:

- what sources exist in this course;
- what has been parsed, indexed, segmented, reconciled, excluded, or used;
- which concepts and block types appear where;
- which source ranges can become notes or review projections;
- which generated projections are based on which material.

Course Material Library is also the answer to a heavy Agent Memory problem. Agent Memory should remember user preferences, not course knowledge. Course knowledge should live in structured material assets and summaries.

## 4. Material Reconciliation Layer

Material Reconciliation is the course-level layer between raw source extraction and canonical NoteBlocks.

Recommended flow:

```text
Upload / Import
  -> Original Source
  -> Normalized Source Snapshot
  -> SourceFragment
  -> MaterialSegment
  -> Material Reconciliation
  -> Canonical NoteBlock
  -> Projection Snapshot
```

Core principles:

```text
Raw source order is preserved, but learning order is reconstructed.
Duplicate knowledge is merged, but source evidence is retained.
Evidence list first, evidence interpretation later.
```

Material Reconciliation should support:

- multi-batch uploads such as `7-9 -> 1-3 -> 4-6`;
- material map proposals that recommend learning order;
- duplicate detection across sources;
- canonical block creation;
- evidence lists for repeated theorem/formula/definition/example mentions;
- missing-bridge candidates;
- scope selection by chapter, week, part, section, page range, or topic cluster;
- manual user reorder, exclusion, restoration, split, and unmerge.

Early versions should not try to rank every source by authority or detail. A theorem mentioned in many sources can first expose an evidence list. Deeper evidence interpretation can come later.

## 5. MaterialSegment And Scope Selection

A `MaterialSegment` is a selectable learning range in a Course Material Library.

Possible segment types:

- chapter;
- week;
- lecture;
- part;
- section;
- page range;
- topic cluster;
- user-selected source range.

This lets users generate notes by scope instead of turning an entire course archive into one giant note.

Example workflow:

```text
User uploads a full-semester note set.
Coincides detects chapters/sections/topics.
User asks to split it into 13 weeks.
Coincides proposes a Study Scope Plan.
User generates only Week 1 now.
Week 2 can be generated later from the same library.
```

This reduces context pressure, improves user control, and makes note generation feel like a staged learning process rather than a one-shot document summary.

## 6. Adaptive Layered Summaries

Summaries should not be based only on file length. Long files can contain dense important sections, and short files can be shallow. Summary depth should be driven by structure, density, importance, and concept coverage.

Useful layers:

- **Source Overview Summary**: what a source is and what it broadly covers.
- **Structural Summary**: chapters, sections, weeks, parts, and detected boundaries.
- **Concept Coverage Summary**: which concepts appear where and in what form.
- **MaterialSegment Summary**: what a selectable learning range contains.
- **NoteBlock Library Summary**: inventory of definitions, theorems, formulas, examples, exercises, diagrams, and source references.
- **Projection Summary**: what a generated note or review set covers and which material it used.

Rule of thumb:

```text
Same importance, same summary depth.
```

Summaries must track dependencies. If source material, segments, blocks, projections, or user exclusions change, dependent summaries should become stale rather than remain silently trusted.

Open question: when should summaries auto-refresh versus wait for a user-triggered refresh or proposal?

## 7. Source Snapshot Viewer

The earlier PDF Reader idea should become `Source Snapshot Viewer`.

It is not a PDF editor. It is a unified evidence viewer and source locator.

Target direction:

```text
Original Source
  -> Normalized Source Snapshot
  -> page-level source reference
```

A source snapshot may be page images plus optional text layer. This supports PDF, Word, scans, images, slides, and mixed materials without building separate readers for every format.

Early scope:

- source preview;
- page jump;
- page-level source reference from NoteBlock or Projection;
- page/page-range selection for import scope;
- lazy snapshot generation for large sources.

Out of early scope:

- full PDF editing;
- required bbox highlight;
- always converting entire textbooks to high-resolution images;
- arbitrary annotation system.

## 8. Typed Proposal System

Proposal is not only a UI modal. It should become Coincides' safety execution protocol.

Possible proposal types:

- `material_map_proposal`;
- `scope_plan_proposal`;
- `source_import_proposal`;
- `note_block_merge_proposal`;
- `organized_note_proposal`;
- `review_projection_proposal`;
- `cross_course_link_proposal` later.

Each proposal should carry affected objects, preview payload, source evidence, confidence where useful, user edits, apply behavior, and rollback expectations where feasible.

This matters for human review and future multi-agent orchestration.

## 9. State, Trash, And Recovery

Course material objects need explicit state.

Candidate processing states:

```text
uploaded
parsed
indexed
segmented
reconciled
ready_for_note
used_in_projection
needs_review
failed
stale
```

Deletion should be status-based:

```text
active -> trashed -> permanently_deleted
```

Trash should not be a separate storage universe. It should preserve relationships and support restore until the user permanently deletes the item.

Material Reconciliation must eventually support recovery actions:

- unmerge sources;
- split canonical block;
- restore excluded material;
- reorder segments;
- undo applied proposal where feasible.

---

## 10. Projection Snapshot Direction

Generated projections should be stable.

If a Week 1 note is generated from selected blocks, later changes to the underlying NoteBlocks should not silently mutate that note.

A projection snapshot may record:

- generated content;
- source block ids;
- source block versions;
- source references;
- selected material segments;
- generated_at;
- proposal/apply history.

If underlying material changes, the user can be offered regenerate, compare, or ignore. Silent drift should be avoided.

## 11. Concept And ConceptMention

Tags alone are not enough for concept-centered workflows.

A tag is good for user organization, such as `exam`, `important`, `chapter 4`, or `midterm`. Concepts like `gradient` need richer representation because they may appear under many forms:

- gradient;
- grad;
- nabla f;
- `\nabla f`;
- gradient vector;
- partial derivatives;
- 梯度.

The first practical version should be course-local. Cross-course concept linking can come later as a candidate/proposal system, not an automatic global merge.

ConceptMention enables workflows such as:

```text
User asks: I want to review gradient.
  -> find course-local Concept(gradient)
  -> collect mentions from source fragments, notes, examples, formulas, diagrams, and projections
  -> propose a Focus Note / Review Set / Practice Set
  -> user reviews and applies
```

## 12. Package Studio / Learning Package Editor

Package Studio is a future auxiliary creation tool for Learning Packages. It should not be treated as a prerequisite for v2.0 or v2.1.

Its first audience is the project creator and internal development workflow. It should help create official Math Pack, default Style Packs, example notes, and package schemas without hand-editing JSON for every experiment.

A Learning Package may eventually include:

- Style Pack / Theme Pack;
- Block Vocabulary;
- Layout Recipe;
- Extraction / Routing Rules;
- Review Projection Rules;
- Example Notes / Test Cases;
- Package Preview / Validator;
- package metadata.

Recommended sequencing:

- v2.0: NoteBlock Foundation.
- v2.1: AI Note Proposal and Course Material Library seed.
- v2.2: Material Reconciliation and package schema primitives.
- v2.3: Source Snapshot Viewer and scoped material selection.
- v2.4+: Package Studio Lite and broader package tooling.

## 13. Product Language

Coincides should use factual, non-judgmental language.

Preferred language:

- `I found these materials.`
- `I organized these structures.`
- `This concept appears in these places.`
- `These blocks can become a review set.`
- `This proposal needs your review before applying.`

Avoid default diagnostic or controlling language:

- `You are weak at this.`
- `You must learn this first.`
- `You are behind.`
- `This is the correct path.`

Coincides is a learning partner and learning steward. It should help users see, organize, and reuse their own material without taking away their agency.

## 14. Migration And Reuse Judgment

The project should not protect the current Card/Deck architecture merely because it exists.

Useful things to inherit from v1.x:

- KaTeX rendering experience;
- FSRS and review scheduling experience;
- Proposal -> Review -> Apply discipline;
- source link and document traceability ideas;
- tag, section, and grouping UX experience;
- batch generation and batch review interactions;
- AI proposal editing and apply flow;
- course/deck/card/document relationship lessons.

Things that may be heavily changed or discarded:

- Card as the primary learning material table;
- Deck as the only collection model;
- old Card content JSON templates;
- the assumption that review units and note units are separate roots;
- the assumption that cards exist without a broader note/block context.

Possible migration direction:

- convert existing Cards into NoteBlocks where possible;
- represent old Decks as legacy review projections or imported review sets;
- keep compatibility UI only if it helps users transition;
- allow the new Course Material Library and NoteBlock Library to become canonical over time.

## 15. Open Questions

- What is the smallest useful block type set for v2.0?
- Should current Cards be migrated into NoteBlocks immediately in v2.0, or only when the new review projection system exists?
- What is the minimum safe ProjectionSnapshot behavior for v2.0?
- How much SourceReference should be real in v2.0 versus reserved for v2.1?
- When should summaries auto-refresh versus wait for user-triggered refresh?
- How should concept aliases handle symbols, English, Chinese, and LaTeX forms?
- What thresholds should Material Scale Router use for small, medium, large, and textbook-scale materials?
- When should `rerank_model` move from optional future role to active retrieval infrastructure?
- Which source snapshot format gives the best tradeoff between speed, storage, and source fidelity?
- Which recovery actions are mandatory before material reconciliation can be trusted?

## 16. Current Working Position

The current working position is:

1. Workspace / Local Profile is the global user root.
2. Course is the learning-domain root.
3. Course Material Library is a first-class Course asset.
4. Agent Memory stores user preferences; Course Material Library stores learning content.
5. v1.x Card/Deck is the ancestor of v2.x NoteBlock Library.
6. v2.x NoteBlock Library should become the canonical learning material substrate.
7. Notes, cards, review sets, exercise sets, formula sheets, and concept focus notes should be projections over that substrate.
8. AI should organize and propose, not silently decide or overwrite.
9. Source evidence and canonical blocks must remain separate.
10. Generated projections should be stable snapshots.
11. Summaries should be adaptive, layered, dependency-aware, and stale-able.
12. Source Snapshot Viewer is the generalization of PDF Reader Lite.
13. Typed Proposal is Coincides' safety execution protocol.
14. Cross-course concept linking is long-term and should not enter the v2.0-v2.2 mainline.
