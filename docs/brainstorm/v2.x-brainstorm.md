# Coincides v2.x Brainstorm

> Status: Brainstorm draft / 草稿池
> Purpose: Record early product and architecture thinking for the v2.x NoteBlock Library, note organization, review projections, concept-focused learning material views, and future migration from the v1.x Card/Deck model.
> Scope note: This document is not the formal roadmap. Mature ideas can later be promoted into `Coincides-Roadmap.md`, `docs/v2-note-system-draft.md`, release plans, or ADRs.

---

## 1. Working Thesis

Coincides v2.x should move from a card-centered learning system toward a block-centered learning material system.

The current v1.x Card/Deck system should be understood as an early prototype of structured learning blocks, not as a permanent peer of the future Note system. It has already validated several important ideas:

- learning material can be split into reusable structured units;
- units can have semantic types such as definition, theorem, formula, and example;
- units can be reviewed, organized, tagged, and generated by AI through Proposal -> Review -> Apply;
- source links and LaTeX rendering matter for academic study workflows.

However, the v1.x Card model is too narrow for the next stage. It lacks full note order, transitions, paragraph context, diagram placement, answer separation, proof flow, page layout, and relationship between blocks. In v2.x, the core should become a `NoteBlock Library`.

The product should still offer something users recognize as a card/review system, but that system should become an upper-layer projection over the NoteBlock Library rather than the primary data source.

## 2. Generational Model

### 2.1 v1.x Card/Deck: Early Prototype

The current Card/Deck system is the ancestor of the future NoteBlock system.

Cards currently hold limited structured academic content:

- definitions;
- theorems;
- formulas;
- examples;
- general notes.

Decks and sections provide a first attempt at grouping these objects. This is useful, but it is not yet a real note system:

- there is no complete title/paragraph/diagram/answer/proof structure;
- there is no natural reading order across all blocks;
- there are no transitions or explanations between blocks;
- examples, answers, diagrams, and formulas are not fully independent fragments;
- a deck is a collection, not a reconstructed learning narrative;
- a card is optimized around review, not around preserving the professor's or student's note structure.

The v1.x system should be treated as a learning experiment and migration source, not a structure that must be protected at all costs.

### 2.2 v2.x NoteBlock Library: Core Substrate

The NoteBlock Library should become the bottom layer for learning materials.

It stores all reusable fragments extracted from user materials, OCR output, textbooks, handwritten notes, AI organization, and user edits. Candidate block types include:

- title;
- heading;
- paragraph;
- definition;
- theorem;
- proof;
- proof step;
- formula;
- example;
- answer;
- exercise;
- question;
- diagram crop;
- image crop;
- source excerpt;
- sidenote;
- annotation;
- divider;
- summary;
- mistake / weak point.

Every block should be source-aware and editable. It should preserve enough context to be reused later in notes, review sets, exercise sets, concept views, and future knowledge maps.

Possible metadata:

- `course_id`;
- `source_document_id`;
- `source_page`;
- `source_bbox`;
- `raw_text`;
- `image_crop_path`;
- `source_order`;
- `confidence`;
- `block_type`;
- `concept_mentions`;
- `created_by` / `updated_by`;
- `operation_batch_id`;
- version or history metadata.

### 2.3 v2.x New Card/Review System: Projection Layer

The future Card/Review System should not simply extend the old Card table. It should be a new upper-layer function that selects, groups, and renders blocks from the NoteBlock Library.

Users may still experience this as a card system, but the source of truth should be the NoteBlock Library.

Examples of review projections:

- Review Card Set;
- Knowledge Checklist;
- Exercise Set;
- Example Set;
- Formula Sheet;
- Theorem-Proof List;
- Proof Practice Set;
- Exam Review Set;
- Mistake / Weak Point Set.

A block can appear in multiple projections. For example, an `example` block can be part of an organized note, an exercise set, a concept focus note, and a review set.

AI may propose review projections, but the user should approve them before they become official.

## 3. Product Architecture: Library and Projections

A useful mental model:

```text
Source Material
  -> SourceFragment
  -> FragmentClassifier
  -> BlockRouter
  -> NoteBlock Library
  -> Views / Projections
```

Upper-layer views and projections may include:

- Organized Notes;
- Review Card Sets;
- Exercise Sets;
- Formula Sheets;
- Theorem-Proof Lists;
- Concept Focus Notes;
- Scoped Knowledge Maps;
- Export Views.

The NoteBlock Library is the shared substrate. Views should not duplicate learning material as unrelated content unless there is a clear versioning or snapshot reason.

This allows the same underlying material to be read, reviewed, reorganized, traced back to source, and remixed by concept.

## 4. AI Note Organization Pipeline

AI should not convert OCR directly into a final note in one opaque step. That would make errors hard to inspect and hard to correct.

Recommended staged flow:

```text
Source Material
  -> OCR / document parsing
  -> SourceFragment
  -> FragmentClassifier
  -> BlockRouter
  -> NoteBlock Proposal
  -> User Review
  -> NoteBlock Library
  -> Organized Note / Review Projection
```

### 4.1 SourceFragment

A SourceFragment is a raw or lightly processed piece of source material. It should preserve evidence from the user's original upload before semantic interpretation.

Examples:

- text region;
- formula-like region;
- diagram or image crop;
- handwritten answer region;
- possible heading;
- possible example area;
- possible proof area;
- side note;
- arrow / divider / annotation.

Suggested metadata:

- document id;
- page;
- bounding box;
- raw OCR text;
- image crop path;
- source order;
- parser/OCR confidence;
- extraction method.

### 4.2 FragmentClassifier

The FragmentClassifier identifies what a fragment might be. It can use deterministic rules for obvious cases and AI for ambiguous cases.

Examples:

- large bold text or isolated top line -> heading candidate;
- math-heavy region -> formula candidate;
- text starting with “Definition” -> definition candidate;
- numbered problem statement -> exercise/example candidate;
- handwritten graph or function sketch -> diagram crop candidate.

### 4.3 BlockRouter

The BlockRouter turns classified fragments into proposed NoteBlocks.

It should decide semantic block type, not final visual style. Styling and layout should happen later through layout recipes, style packs, and user editing.

Early block types should be limited enough to ship safely:

- heading;
- paragraph;
- definition;
- theorem;
- proof;
- formula;
- example;
- answer;
- exercise;
- diagram_crop;
- image_crop;
- sidenote;
- divider.

### 4.4 Proposal First

AI-created notes, blocks, review sets, and concept focus notes should follow Proposal -> Review -> Apply.

The user should be able to:

- accept proposed blocks;
- edit block text;
- change block types;
- split or merge blocks;
- hide or delete blocks;
- move blocks;
- inspect source evidence;
- reject the proposal without changing official data.

## 5. Organized Notes

Organized Notes are one projection over the NoteBlock Library.

They are not the only purpose of blocks. They are the readable, page-oriented arrangement of selected blocks.

Important direction:

- support A4 / A5 / Letter / custom fixed-page formats first;
- defer infinite canvas until there is a clear need;
- preserve classroom context and professor/student note style where useful;
- make every block user-editable;
- allow manual rearrangement after AI proposal;
- preserve original diagrams as crops before attempting AI redrawing.

Different subjects may need different layout styles:

- history/social science notes may be mostly linear text and headings;
- calculus notes may need formulas, examples, graphs, theorem/proof flows, and diagram crops;
- biology notes may need diagram annotation and image-heavy blocks;
- organic chemistry notes may need reaction/structure-oriented layouts.

This is why the system should eventually support layout recipes and style packs, but the first version should avoid a fully general template engine.

## 6. Review and Card Projection

The future review/card system should be rebuilt around NoteBlocks.

A user should be able to select important blocks from an organized note and turn them into a review set. Selection may happen through:

- clicking checkboxes;
- block multi-select;
- drag selection;
- manual add-to-review actions;
- AI proposal based on selected notes or concepts.

AI may suggest a review projection such as:

- “key formulas from this chapter”;
- “all examples involving gradient”;
- “theorem and proof list for exam review”;
- “exercise set from uploaded textbook section”;
- “concept checklist for calculus 3 midterm.”

The user can then apply, edit, regenerate one item, regenerate a batch, retag, reorder, or discard.

This keeps the familiar card/review experience while changing the underlying source of truth.

## 7. Concept and ConceptMention

Tags alone are not enough for concept-centered workflows.

A tag is good for user organization, such as `exam`, `important`, `chapter 4`, or `midterm`. But concepts like `gradient` need a richer representation because they may appear under many forms:

- gradient;
- grad;
- nabla f;
- `\nabla f`;
- gradient vector;
- partial derivatives;
- 梯度.

Recommended direction:

```ts
type Concept = {
  id: string;
  course_id: string;
  canonical_name: string;
  aliases?: string[];
};

type ConceptMention = {
  id: string;
  concept_id: string;
  source_type: 'source_fragment' | 'note_block' | 'document_chunk' | 'review_projection';
  source_id: string;
  block_type?: string;
  page?: number;
  context_excerpt?: string;
  confidence?: number;
};
```

The first practical version can be course-local. Cross-course concept linking can come later.

ConceptMention enables workflows such as:

```text
User asks: “I want to review gradient.”
  -> find Concept(gradient)
  -> collect related mentions from notes, textbook chunks, examples, formulas, diagrams, and review projections
  -> propose a Focus Note / Review Set / Practice Set
  -> user reviews and applies
```

The language should stay factual. Coincides should say things like “gradient appears in these examples and formulas,” not “you are weak at gradient,” unless the user explicitly asks for diagnostic judgment.

## 8. Focus Note / Concept Note

A Focus Note is a generated or user-curated projection centered around one concept.

Example: a user studying Waterloo MATH 237 wants to review `gradient`.

The system could gather:

- definitions of gradient from the user's notes;
- textbook excerpts that mention gradient;
- formulas involving gradient;
- examples using gradient;
- diagrams or graph crops related to gradient;
- theorem/proof blocks involving gradient;
- exercises or answers where gradient appears;
- existing review projections related to gradient.

The output should be a proposal, not an automatic final note.

Possible outputs:

- Focus Note;
- Review Card Set;
- Exercise Set;
- Formula Sheet;
- Concept Checklist.

This is a natural extension of the NoteBlock Library: the user is not asking the AI to teach from nowhere; they are asking Coincides to reorganize their own learning materials around a concept.

## 9. Migration and Reuse Judgment

The project should not protect the current Card/Deck architecture merely because it exists.

The target product experience matters more than preserving v1.x structures.

Useful things to inherit from v1.x:

- KaTeX rendering experience;
- FSRS and review scheduling experience;
- Proposal -> Review -> Apply discipline;
- source link and document traceability ideas;
- tag, section, and grouping UX experience;
- batch generation and batch review interactions;
- AI proposal editing and apply flow;
- course/deck/card/document relationship lessons.

Things that may be heavily changed or discarded:

- Card as the primary learning material table;
- Deck as the only collection model;
- old Card content JSON templates;
- the assumption that review units and note units are separate roots;
- the assumption that cards exist without a broader note/block context;
- the current lack of source bbox, block order, block relationships, and layout context.

Possible migration direction:

- convert existing Cards into NoteBlocks where possible;
- represent old Decks as legacy review projections or imported review sets;
- keep compatibility UI only if it helps users transition;
- allow the new NoteBlock Library to become the canonical source over time.

The migration strategy should be decided by product quality, not by attachment to existing implementation work.

## 10. Open Questions

- Should the core name be `NoteBlock`, `LearningBlock`, or another neutral term?
- Should current Cards be migrated into NoteBlocks immediately in v2.0, or only when the new review projection system exists?
- What is the smallest useful block type set for v2.0?
- How much source layout metadata should be required before AI note organization ships?
- Should review projections snapshot block content or always reference live blocks?
- How should a user intentionally fork a block for review wording without changing the original note block?
- How should block-level version history work?
- How should AI split/merge mistakes be corrected by the user?
- How should concept aliases handle symbols, English, Chinese, and LaTeX forms?
- Which subject should be used as the first full NoteBlock test case: calculus, history/social science, biology, or organic chemistry?

---

## 11. Current Working Position

The current working position is:

1. v1.x Card/Deck is the ancestor of v2.x NoteBlock Library.
2. v2.x NoteBlock Library should become the canonical learning material substrate.
3. Notes, cards, review sets, exercise sets, formula sheets, and concept focus notes should be projections over that substrate.
4. AI should organize and propose, not silently decide or overwrite.
5. Source evidence and user editability are required for trust.
6. Existing Card/Deck work is valuable as learning and partial infrastructure, but it should not constrain the final v2 product shape.
