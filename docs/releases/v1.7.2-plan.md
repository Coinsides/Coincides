# Coincides v1.7.2 — 学习计划流程完善：日期选择 + Time Block 缺口处理

> **主题**：学习计划创建流程完善 — 月历日期选择器 + Time Block 缺口检测与补全 + 两步 Proposal 流程  
> **前置版本**：v1.7.1（Agent 效率深度优化）  
> **目标**：解决当前学习计划创建流程中"日期无法精确选择"和"Time Block 覆盖不全导致任务无归属"两个核心问题，使 Agent 生成的学习计划任务都能正确挂载到 Time Block 下。

---

## 一、问题背景

用户在使用学习计划功能时发现以下问题：

1. **日期选择缺失**：当前偏好收集表单 (`collect_preferences`) 没有日期选择字段，Agent 只能猜测用户想在哪些日子学习
2. **Time Block 覆盖不全**：用户可能只设置了部分星期的 Time Block（如周一到周五），但选择了包含周末的日期范围，导致周末任务无 `time_block_id`
3. **任务无归属**：`time_block_id` 未填充的任务在日历上"悬空"，不显示在对应 Time Block 区域内

### 根因分析

- Time Block 是周模板系统（`day_of_week` 0-6），而非按日期设置
- `time_block_overrides` 表支持按日期调整，但 Agent 不具备"检测缺口 → 提议补全"的流程
- 偏好收集表单缺少 `date_picker` 类型的问题控件

---

## 二、设计宪法守则

| 宪法条款 | v1.7.2 约束 |
|---|---|
| 不替用户做决定 | Time Block 缺口由 Agent 检测，但补全方案必须通过 Proposal 让用户审批；Agent 不自动创建 Time Block |
| 不监控用户 | 不根据用户已有 Time Block 推断学习习惯或自动建议优化 |
| 不制造挫败感 | 缺口检测结果以中性语气呈现（"这几天还没有设置 Time Block"），不暗示用户做得不对 |

---

## 三、核心设计决策（已确认）

### 3.1 日期选择：月历视图组件

- 新增 `date_picker` 类型的偏好表单控件
- UI 为月历视图（MonthCalendar），支持两种选择模式：
  - **拖选**：选中连续日期范围（如 3月22日 到 3月28日）
  - **单击**：选中/取消单个日期（如只选周一三五）
- 返回排序后的日期数组：`["2026-03-22", "2026-03-23", ...]`
- 不使用自由文本输入，不使用结构化日期输入（如分别输入起止日期）

### 3.2 Time Block 缺口处理：两步 Proposal

当用户选择的日期中存在 Time Block 缺口时，流程变为：

```
Step 1: Agent 调用 get_time_blocks(templates_only) 获取周模板
Step 2: 对比用户选中日期的 day_of_week 与已有模板，识别缺口
Step 3: 如有缺口 → 生成 Time Block Proposal（补全缺失日的 Time Block）
Step 4: 用户审批 Time Block Proposal
Step 5: 生成 Study Plan Proposal（所有任务都有 time_block_id）
Step 6: 用户审批 Study Plan Proposal
```

如果无缺口，跳过 Step 3-4，直接生成 Study Plan Proposal。

### 3.3 Time Block 是周模板

- `time_blocks` 表用 `day_of_week`（0=周日, 1=周一 ... 6=周六）定义
- 一次补全创建的 Time Block 会应用到每周该日，不是只针对单次
- `time_block_overrides` 表可按日期微调（但 v1.7.2 不涉及 override 的自动创建）

### 7.3 Time Block Gap Detection 协议

新增 System Prompt 段落：

```
## Time Block Gap Detection（Time Block 缺口检测）

在收到用户的偏好表单响应后（包含 study_dates），执行以下检测：

1. 调用 get_time_blocks(templates_only=true) 获取现有周模板
2. 从 study_dates 提取所有涉及的 day_of_week 值
3. 对比：哪些 day_of_week 没有 Study 类型的 Time Block

### 无缺口
直接进入 Study Plan Proposal 生成。

### 有缺口
生成 time_block_setup Proposal：
- 向用户说明："你选择的日期中，[周X、周X] 还没有设置学习时间段。我建议先补充这些天的 Time Block。"
- Proposal 包含缺失日的 Time Block 建议（参考已有模板的时间段，如果已有模板为 8:00-18:00，建议缺口日也使用类似时段）
- 如果完全没有任何 Time Block 模板，提供合理默认值（如 9:00-12:00, 14:00-18:00）
- 用户审批后再生成 Study Plan Proposal

### 注意
- 只检测 Study 类型的 Time Block 缺口，非 Study 类型（如 meal, rest）不纳入缺口判断
- 缺口提示语气中性，不暗示用户做得不对
- 用户可以拒绝 Time Block Proposal → Agent 应告知"这些日期的任务将无法挂载到 Time Block 下，会以普通日历事件形式显示"，然后继续生成 Study Plan
```

### 7.4 time_block_setup Proposal 格式

{
  "type": "time_block_setup",

  "description": "你选择的学习日期包含周六和周日，但这两天还没有设置学习时间段。建议补充以下 Time Block：",
  "items": [

### 8.3 time_block_id 自动填充逻辑

在 `proposals.ts` 的 study_plan apply 阶段增加防御逻辑：

```typescript
// 对每个 task item，如果 time_block_id 为空
if (!item.time_block_id && item.scheduled_date) {
  const date = new Date(item.scheduled_date);
  const dayOfWeek = date.getDay(); // 0=Sunday, 6=Saturday
  
  // 查找该 day_of_week 的 Study 类型 Time Block
  const matchingBlock = db.prepare(`
    SELECT id FROM time_blocks 
    WHERE user_id = ? AND day_of_week = ? AND type = 'study'
    ORDER BY start_time ASC LIMIT 1
  `).get(userId, dayOfWeek);
  
  if (matchingBlock) {
    item.time_block_id = matchingBlock.id;
  }
}
```

这是一层防御性兜底——正常流程中 Agent 应在 proposal items 中提供 `time_block_id`，此逻辑处理 Agent 遗漏的情况。

### 8.4 System Prompt Anti-Pattern 新增

```
| ❌ 禁止行为 | ✅ 正确做法 |
|---|---|
| study_plan item 不含 time_block_id（Time Block 模式下） | 每个 item 根据 scheduled_date 的 day_of_week 查找匹配的 Time Block ID 并填入 |
| 用户选了没有 Time Block 的日期就直接排任务 | 先检测缺口，生成 time_block_setup Proposal 补全后再排 |
```

## 九、关键依赖与风险

| 风险 | 影响 | 缓解方案 |
|------|------|----------|
| MonthCalendar 拖选交互在移动端触控不佳 | 中 | 当前产品为桌面端优先，移动端后续版本处理；单击选择作为 fallback |
| 同一 day_of_week 有多个 Study Time Block | 低 | 缺口检测只看"是否存在至少一个 Study Block"，不判断数量；`time_block_id` 填充取第一个（按 start_time 排序） |
| 用户拒绝 Time Block Proposal 后任务无归属 | 中 | 明确告知用户后果，允许继续（宪法第一条：不替用户做决定）；防御性填充逻辑兜底 |
| 两步 Proposal 流程用户觉得繁琐 | 低 | 用户已确认可接受；无缺口时自动跳过，不增加额外步骤 |
| date_picker 组件增加 PreferenceForm 复杂度 | 低 | MonthCalendar 作为独立组件，PreferenceForm 仅负责集成渲染 |
