# Coincides v1.5 — 版本规划

> **主题**：Time Block 重构：图层嵌套 + 编辑模式 + 任务关联
> **前置版本**：v1.4（Time Block L1 补全 + 卡片数据模型升级）
> **目标**：重新定义 Time Block 的视觉呈现和交互模型。Study Block 作为主层，Meal/Rest/Custom 嵌套叠加；前端交互全面升级（右键菜单创建、编辑面板、编辑模式切换）；Agent 读取可用学习时间制定计划；任务显式关联 Block。

---

## 一、核心概念对齐

v1.4 补全了 Time Block 的可视化基础（刻度线、时间标签、右键编辑/删除），但当前交互和视觉仍有根本性问题：

**交互问题**：
- 拖拽框选后直接弹出内嵌表单，因周视图列宽过窄（~130px），表单被截断、按钮不可见
- 框选误差无法修正——用户拖拽完毕后无法调整起止时间
- 创建 Time Block 的流程和浏览模式混在一起，缺少明确的"编辑态"

**视觉问题**：
- 全局内容区 `max-width: 1000px` + 左右 `44px` padding，日历两侧大量空白浪费
- Time Block 之间只有重叠警告，没有嵌套语义——实际上 Study Block 内嵌 Meal/Rest 是正常场景
- 默认始终显示 0:00–24:00 全天刻度，大部分时间段是空白

**数据模型问题**：
- 任务没有关联到具体 Time Block，Agent 无法知道哪些任务属于哪个学习时段
- 不存在"每天最多 1 个 Study Block"的约束

### 设计理念（Session 3 决策）

| 原则 | 说明 |
|------|------|
| 单 Study Block 约束 | 每天最多 1 个 Study Block，Meal/Rest/Custom 不限 |
| 可用学习时间公式 | Study Block 时长 − 嵌套的非学习 Block 时长 |
| 图层嵌套 | 时间最长的 Block 在最底层，短 Block 内缩显示 |
| 浮空预览 | 默认无网格线，Block 边缘延伸时间标注线 |
| 动态时间范围 | 视图起止 = 当周最早/最晚 Block ± 留白 |
| 编辑模式切换 | 点击按钮进入编辑模式才能创建/修改 Block |
| 右键菜单创建 | 框选后右键触发菜单 → "添加 Time Block" → 编辑面板（含时间选择器） |
| 任务显式关联 | 任务归属具体 Block，Block 内列表展示任务 |

## 二、设计宪法守则（v1.5 特别提醒）

v1.5 涉及 Agent 排期逻辑重写和 UI 交互模式变更，需严格遵守：

| 宪法条款 | v1.5 约束 |
|---|---|
| 不替用户做决定 | Agent 只根据 Study Block 可用时间建议排期，不自动创建/修改 Block |
| 不监控用户 | 不追踪 Block 实际使用时间，不记录"是否按 Block 执行" |
| 不制造挫败感 | 没有设置 Time Block 的日期照常可用（fallback 120min），不提示"你还没设置今天的时间" |

## 四、Step 拆分

### Step 1：全局布局拓宽（TB-R12）

**目标**：去除全局内容区左右多余留白，让日历、统计等页面充分利用屏幕宽度。

**变更**：
1. `AppLayout.module.css` — `.content` 样式调整：
   - 去除或大幅放宽 `max-width: 1000px`（改为 `max-width: 1400px` 或去除限制）
   - 缩减 `padding: 36px 44px` → `padding: 28px 24px`（或更紧凑）
2. 逐页检查：确认 DailyBrief、Calendar、Goals、Courses、Decks、Statistics、Settings 在拓宽后布局正常，无异常拉伸

**验收标准**：
- [ ] 日历周视图每列明显变宽，表单/Block 有更多空间
- [ ] 各页面在 1920px 宽度下无异常拉伸
- [ ] 在 1280px 和 1024px 宽度下仍可正常使用

### Step 2：Time Block 创建交互重构（TB-R11）

**目标**：将"框选 → 直接弹内嵌表单"改为"框选 → 右键菜单 → 编辑面板"。

**变更**：
1. `Calendar.tsx` — 框选行为调整：
   - 框选完成后（mouseup）不再自动弹出 `tbFormPopup`
   - 框选区域保持高亮，等待用户右键触发
   - 右键弹出上下文菜单：「添加 Time Block」选项
   - 点击「添加 Time Block」→ 弹出编辑面板
2. 编辑面板（新建或复用现有 `tbEditModal`）：
   - 标题输入框（Label）
   - 类型选择器（Study / Sleep / Custom）
   - **起始时间选择器**（`<input type="time">`）— 预填框选起始时间
   - **结束时间选择器**（`<input type="time">`）— 预填框选结束时间
   - 颜色选择器（可选）
   - Save / Cancel 按钮
3. `Calendar.module.css` — 移除旧的 `.tbFormPopup` 内嵌表单样式，新增编辑面板样式
4. 面板定位：作为 overlay 居中弹窗，不受列宽限制

**验收标准**：
- [ ] 框选后区域保持高亮，不自动弹表单
- [ ] 右键弹出菜单，点击"添加 Time Block"后打开编辑面板
- [ ] 编辑面板中可修正起止时间（time picker）
- [ ] 面板不受列宽限制，所有元素完整可见
- [ ] ESC 或点击外部关闭面板
- [ ] Enter 快捷提交

### Step 3：编辑模式切换（TB-R5）

**目标**：新增编辑模式开关，只有编辑模式下才能框选创建 Time Block。

**变更**：
1. `Calendar.tsx` — 新增状态 `editMode: boolean`
   - 周视图顶部（或导航栏旁）新增编辑模式切换按钮（铅笔图标 / Edit 图标）
   - `editMode = false`（默认）：禁用框选拖拽，点击列仅查看
   - `editMode = true`：启用框选拖拽 + 右键创建流程，显示网格线
   - 切换时有视觉反馈（按钮高亮 / 边框颜色变化）
2. `Calendar.module.css` — 编辑模式视觉指示（列边框变色、光标变为 crosshair）

**验收标准**：
- [ ] 默认不可框选，必须先点击编辑模式按钮
- [ ] 编辑模式下列边框变色、光标变 crosshair，网格线显示
- [ ] 退出编辑模式后网格线隐藏（除非偏好常开）
- [ ] 编辑模式状态不持久化（刷新后回到默认关闭）

### Step 4：网格线偏好 + 删除重叠警告（TB-R6 + TB-R7）

**目标**：网格线可在非编辑模式下常开（偏好），同时移除重叠警告。

**变更**：
1. `Calendar.tsx` — 读取 `localStorage` 中 `tb-grid-preference`（`'always'` | `'edit-only'`），默认 `'edit-only'`
   - `always`：非编辑模式下也显示网格线
   - `edit-only`：仅编辑模式显示
2. Settings 页面或日历页面内添加网格线偏好开关（简单 toggle）
3. 移除 overlap 检测逻辑：
   - `Calendar.tsx` — 删除 overlap 计算、`overlapSegments` 渲染、`⚠️` 警告图标
   - `Calendar.module.css` — 移除 overlap 相关样式

**验收标准**：
- [ ] 默认仅编辑模式显示网格线
- [ ] 切换偏好为 always 后，非编辑模式也显示网格线
- [ ] 偏好刷新后保持
- [ ] 重叠的 Time Block 不再显示警告图标

### Step 5：Study Block 单层约束（TB-R10）

**目标**：每天最多 1 个 Study Block，Meal/Rest/Custom 不限。

**变更**：
1. **后端** — `POST /api/time-blocks` 和 `PUT /api/time-blocks/:id`：
   - 创建/编辑 type=study 时，检查同 `day_of_week` 是否已有 study Block
   - 有则返回 400 + 错误信息
2. **前端** — 编辑面板中：
   - 选择 type=study 时，若该天已有 Study Block，禁用 Save + 显示提示文字
   - 也可在类型选择器中灰化 Study 选项
3. `Calendar.tsx` — 周视图渲染中，Study Block 如有多个（兼容旧数据）不做崩溃处理，正常显示但标记

**验收标准**：
- [ ] 尝试在同一天创建第二个 Study Block 时，后端返回 400
- [ ] 前端面板显示提示"每天只能有一个 Study Block"
- [ ] Meal/Rest/Custom 不受限制
- [ ] 已有旧数据（如有多个 Study）不会崩溃

### Step 6：图层嵌套渲染（TB-R2）

**目标**：长 Block 作为底层容器，短 Block 在其内部缩进渲染，形成嵌套视觉。

**变更**：
1. `Calendar.tsx` — 嵌套计算算法：
   - 对每天的 Block 按时长降序排列
   - 最长 Block（通常是 Study）渲染为底层，`left: 4px; right: 4px`
   - 嵌套在其内部的短 Block 递增缩进（每层 +8px padding）
   - 嵌套判断条件：短 Block 的 [start, end] 完全落在长 Block 的 [start, end] 内
   - 非嵌套 Block（时间不重叠）独立渲染
2. `Calendar.module.css` — 嵌套层级视觉样式：
   - 底层 Block 背景更淡
   - 嵌套 Block 有左边距缩进 + 略深背景
   - hover 时展开 tooltip 显示完整信息
3. z-index 逻辑：底层 Block z-index 低，嵌套 Block z-index 递增

**验收标准**：
- [ ] Study Block（8:00–18:00）内嵌 Eat Block（12:00–13:00），Eat 在 Study 内部缩进显示
- [ ] 多层嵌套正确渲染（3 层以上不崩溃）
- [ ] 非嵌套 Block 独立渲染不受影响
- [ ] 嵌套视觉层次分明，底层淡色，上层深色

### Step 7：浮空预览 + 动态时间范围（TB-R3 + TB-R4）

**目标**：默认无网格线的浮空预览视图，时间范围根据实际 Block 动态调整。

**变更**：
1. `Calendar.tsx` — 动态时间范围计算：
   - 扫描当周所有 Block，找出最早 start_time 和最晚 end_time
   - 视图范围 = min(start) − 1h ~ max(end) + 1h（留白）
   - 无 Block 的日子：fallback 到 8:00–22:00
   - 时间刻度栏根据动态范围生成
   - `timeStrToPercent()` 改为基于动态范围而非固定 24h
2. `Calendar.module.css` — 浮空预览视觉：
   - Block 边缘延伸水平标注线（细虚线，从 Block 边缘延伸到时间刻度栏方向）
   - 标注线末端显示时间数字（如 `9:00`）
   - 非编辑模式下网格线默认隐藏

**验收标准**：
- [ ] 只有 8:00–18:00 有 Block 时，视图范围约 7:00–19:00，不显示凌晨
- [ ] 完全没有 Block 的一天显示 8:00–22:00 默认范围
- [ ] Block 边缘有标注线延伸到刻度栏
- [ ] 动态范围变化时刻度栏同步更新
- [ ] 现有 Task 事件定位不受影响（也基于动态范围重算百分比）

---

### Step 8：任务显式关联 Time Block（TB-R8）

**目标**：任务与具体 Time Block 建立关联，Block 内以列表形式展示所属任务。

**变更**：
1. **数据库** — Migration：
   - `tasks` 表新增 `time_block_id TEXT REFERENCES time_blocks(id) ON DELETE SET NULL`
   - 新增索引 `idx_tasks_time_block ON tasks(time_block_id)`
2. **后端** — API 更新：
   - `POST /api/tasks` 和 `PUT /api/tasks/:id` 支持 `time_block_id` 字段
   - `GET /api/tasks` 返回中包含 `time_block_id`
   - 新增 `GET /api/time-blocks/:id/tasks` — 获取某 Block 下所有关联任务
3. **前端** — Block 内任务列表：
   - `Calendar.tsx` — 每个 Block 底部渲染关联任务数量 badge
   - 点击 Block 展开任务列表（或 hover tooltip）
   - 任务列表显示标题 + 状态 + 优先级
4. **前端** — 任务创建/编辑：
   - TaskModal 新增可选的 Time Block 关联选择器
   - Agent 创建任务时可指定 `time_block_id`
5. **shared/types.ts** — Task 类型新增 `time_block_id?: string`

**验收标准**：
- [ ] 创建任务时可选择关联 Time Block
- [ ] Block 上显示关联任务数量
- [ ] 点击/hover Block 能看到任务列表
- [ ] 删除 Block 后，关联任务的 time_block_id 自动置空（ON DELETE SET NULL）
- [ ] Agent 创建任务时能指定 time_block_id

### Step 9：Agent 可用时间计算重写（TB-R9）

**目标**：Agent 排期引擎正确计算可用学习时间 = Study Block − 嵌套非学习 Block。

**变更**：
1. `scheduling.ts` — `getDailyCapacities()` 重写：
   - 取当天所有 Block
   - 找到 Study Block（最多 1 个）→ 基础学习时间
   - 找到嵌套在 Study Block 内的非学习 Block → 排除时间
   - 可用学习时间 = Study Block 时长 − Σ(嵌套非学习 Block 时长)
   - 没有 Study Block 的日子：fallback 120min（现有逻辑）
2. `definitions.ts` — Agent 工具描述更新：
   - `schedule_tasks` 描述中说明"可用时间基于 Study Block 减去嵌套 Block"
   - `create_task` 工具描述新增 `time_block_id` 参数
3. `system-prompt.ts` — 更新排期相关指导文字

**验收标准**：
- [ ] Study Block 8:00–18:00 内嵌 Eat 12:00–13:00 → 可用时间 = 540min（9h），不是 600min（10h）
- [ ] 没有 Study Block 时 fallback 120min
- [ ] Agent 排期任务时正确考虑嵌套扣减
- [ ] Agent 创建任务时可指定 time_block_id

## 五、技术风险

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 动态时间范围 → 现有 Task 事件定位错误 | Task 的 start_time/end_time 百分比按 24h 计算，改为动态后需全部重算 | `timeStrToPercent()` 统一改为接收动态范围参数 |
| 图层嵌套算法性能 | 每天 Block 数量较少（通常 1-5 个），O(n²) 嵌套检测可接受 | 如 Block 数量异常多，限制嵌套深度为 5 层 |
| Study Block 单层约束 → 旧数据不兼容 | 用户可能已创建多个 Study Block | 后端仅校验新建/编辑，旧数据保留不删除，前端正常渲染 |
| 右键菜单在 Electron 中行为 | Electron 可能拦截 right-click | 测试 Electron 环境，必要时用 `e.preventDefault()` |
| tasks.time_block_id 迁移 | 旧任务无关联 Block | 新增字段可空，现有数据不受影响 |
| 全局拓宽 → 部分页面过宽 | DailyBrief 等文本密集页面可能阅读不适 | 可按页面类型设置不同 max-width（日历/统计类宽屏，brief/设置类中等） |
