# Coincides v1.3 — 版本规划

> **主题**：工作流引擎 + Time Block + L1 入驻流 + 设计宪法审计
> **前置版本**：v1.2（AI 交互重构 + Goal Manager + 新用户引导 + i18n）
> **目标**：将 v1.2 的引导骨架升级为完整的「大目标 → 子目标 → Task → 日历排期」自动化工作流，引入 Time Block 周模板系统，实现会议记录中定义的 L1 入驻流全链路，并对全产品进行设计宪法合规审计。

---

## 一、核心概念对齐

v1.2 完成了 AI 脚手架重写 + Goal Manager 树形 UI + 引导骨架 + i18n，但缺少以下关键闭环：

- **工作流引擎**：v1.2 的 Agent 已能拆解目标为子 Goal + Task，但生成的 Task 没有时间段、没有排期能力。工作流引擎 = Agent 拆解 + **自动排期** + 执行追踪的完整闭环。
- **Time Block**：用户每天的时间结构（学习、睡觉、娱乐等）目前无处可存。Time Block 是周级模板系统，代表用户默认的时间节奏，AI 据此计算可用学习时长和排期约束。
- **L1 入驻流**：v1.2 的 Onboarding 是 3 步 UI 引导（创建课程 → 上传材料 → 打开 AI），但没有完成会议记录中定义的完整 L1 流程（收集参数 → AI 拆解 → 生成 Proposal → 用户审批 → Task 落到日历）。
- **设计宪法审计**：v1.2 将三条原则硬编码进 System Prompt，但产品的其他模块（DailyBrief、Statistics、Calendar）是否也符合宪法？需要全面审计。

> v1.3 完成后，新用户从注册到第一份学习计划落地日历，全链路自动化。老用户通过 Time Block 维护个人时间节奏，AI 据此安排任务。

## 三、数据模型设计

### 新增表：`time_blocks`

```sql
CREATE TABLE IF NOT EXISTS time_blocks (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  label TEXT NOT NULL,                   -- '学习', '睡觉', '娱乐', 自定义
  type TEXT NOT NULL DEFAULT 'custom',   -- 'study' | 'sleep' | 'custom'
  day_of_week INTEGER NOT NULL,          -- 0=周日, 1=周一, ..., 6=周六
  start_time TEXT NOT NULL,              -- 'HH:MM' 格式
  end_time TEXT NOT NULL,                -- 'HH:MM' 格式
  color TEXT,                            -- 可选颜色，默认按 type 自动分配
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_time_blocks_user ON time_blocks(user_id);
CREATE INDEX IF NOT EXISTS idx_time_blocks_user_day ON time_blocks(user_id, day_of_week);
```

### 新增表：`time_block_overrides`

```sql
CREATE TABLE IF NOT EXISTS time_block_overrides (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  time_block_id TEXT NOT NULL REFERENCES time_blocks(id) ON DELETE CASCADE,
  override_date TEXT NOT NULL,            -- 'YYYY-MM-DD'，覆盖哪一天
  start_time TEXT,                        -- NULL = 该天删除此 Block
  end_time TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_tbo_user_date ON time_block_overrides(user_id, override_date);
```

### 新增表：`goal_dependencies`

```sql
CREATE TABLE IF NOT EXISTS goal_dependencies (
  id TEXT PRIMARY KEY,
  goal_id TEXT NOT NULL REFERENCES goals(id) ON DELETE CASCADE,
  depends_on_goal_id TEXT NOT NULL REFERENCES goals(id) ON DELETE CASCADE,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  UNIQUE(goal_id, depends_on_goal_id)
);

CREATE INDEX IF NOT EXISTS idx_goal_deps_goal ON goal_dependencies(goal_id);
CREATE INDEX IF NOT EXISTS idx_goal_deps_depends ON goal_dependencies(depends_on_goal_id);
```

### 设计说明

**Time Block**：
- `time_blocks` 是**周模板**：每条记录代表「每周 X 的 HH:MM-HH:MM 是什么类型的时间」
- `time_block_overrides` 是**单日覆盖**：某一天需要不同的安排时，覆盖模板（start_time/end_time 为 NULL 表示该天删除此 Block）
- `type = 'study'` 是 AI 排期的核心输入：Agent 只在 study 类型 Block 内安排 Must Task
- 不需要填满 24 小时——未分配的时间 = 用户自由支配，系统不过问（设计宪法第二条）
- **重叠允许存在**：两个 Time Block 时间重叠时不阻止保存，只做视觉提示（叠加混合色 + ℹ️ 图标）
- 不创建 `events` 表——Task 本身就是日历事件（v1.1 已确认，meeting notes §9.1）

**Goal 依赖**：
- `goal_dependencies` 是**中间表**，天然支持 DAG（一个 Goal 可依赖多个 Goal）
- **v1.3 只做线性链**：UI 限制每个 Goal 最多选一个前置依赖，但数据库不限制
- 未来版本做多依赖时，**只改 UI 和 Agent 逻辑，不动数据库**
- 环检测必须在创建依赖时执行（DFS），防止循环依赖

## 四、Step 拆分

### Step 1：Time Block + Goal 依赖 数据模型 + CRUD（TB-1 + TB-2 + GD-1 + GD-4）

**目标**：建立 Time Block 周模板系统 + Goal 依赖链的后端基础。

**工作内容**：

**Time Block**：
- Migration `007_time_blocks.ts`：创建 `time_blocks` 和 `time_block_overrides` 表
- `server/src/routes/timeBlocks.ts`：完整 CRUD API
  - `GET /api/time-blocks` — 获取用户所有 Time Block（周模板）
  - `POST /api/time-blocks` — 创建时间块（支持批量创建：传入数组）
  - `PUT /api/time-blocks/:id` — 修改时间块
  - `DELETE /api/time-blocks/:id` — 删除时间块
  - `GET /api/time-blocks/day/:date` — 获取某天的实际 Time Block（模板 + Override 合并）
  - `GET /api/time-blocks/week/:date` — 获取某周的实际 Time Block（7 天合并结果）
  - `POST /api/time-blocks/override` — 创建单日覆盖
  - `DELETE /api/time-blocks/override/:id` — 删除单日覆盖
- 辅助函数：`getAvailableStudyMinutes(userId, date)` — 计算某天的可用学习分钟数
- 辅助函数：`detectOverlaps(blocks)` — 检测重叠 Time Block（返回重叠对，不阻止保存）

**Goal 依赖**：
- Migration `008_goal_dependencies.ts`：创建 `goal_dependencies` 表
- `server/src/routes/goals.ts` 新增：
  - `POST /api/goals/:id/dependencies` — 添加依赖（含 DFS 环检测）
  - `DELETE /api/goals/:id/dependencies/:depId` — 删除依赖
  - `GET /api/goals/:id/dependencies` — 获取某 Goal 的依赖列表
- `list_goals` API 返回值新增 `dependencies` 字段
- DFS 环检测函数：`detectCycle(goalId, newDependsOnId)` — 添加依赖前检查是否形成环

**共通**：
- `shared/types/index.ts`：新增 TimeBlock、TimeBlockOverride、GoalDependency 接口
- `schema.sql` 同步更新

**验收标准**：
- [ ] Time Block CRUD API 全部正常工作（含批量创建）
- [ ] 获取某天/某周的 Time Block 正确合并模板和 Override
- [ ] `getAvailableStudyMinutes` 正确计算（只统计 type='study' 的 Block）
- [ ] `detectOverlaps` 正确返回重叠对
- [ ] 删除 Time Block 时级联删除对应 Override
- [ ] Goal 依赖 CRUD 正常工作
- [ ] 环检测：A→B→C 后再添加 C→A 被拒绝
- [ ] 删除 Goal 时级联删除其依赖记录

### Step 3：Agent 排期引擎（WF-1 ~ WF-5 + TB-5 + GD-3）

**目标**：Agent 理解用户时间结构 + Goal 依赖关系，能自动排期并支持增量重排。

**工作内容**：

**Agent 新工具**：
- `get_time_blocks`：返回周模板 + 指定日期范围的实际 Time Block + 每日 `available_study_minutes`
- `get_goal_dependencies`：返回指定 Goal 的依赖链（含递归：A→B→C 返回完整顺序）

**Agent 工具升级**：
- `get_tasks`：返回值新增已有 Task 的时间段占用 + Task 状态（pending/completed）
- `list_goals`：返回值新增 `dependencies` 字段 + 各 Goal 的完成进度

**排期引擎**：
- 新增 `scheduleTasksAcrossDays()`：
  - 输入：Task 列表 + 截止日期 + 每日可用学习分钟数 + 已有 Task 占用 + Goal 依赖顺序
  - 输出：每个 Task 分配到具体的 date（不锁到具体时间点，符合设计宪法第三条）
  - 逻辑：
    1. 按 Goal 依赖顺序排序：前置 Goal 的 Task 先排
    2. Must 优先排满，然后 Recommended，最后 Optional
    3. 每天 Must 不超过 study 时长
    4. 贪心策略：截止日期近的优先

**增量重排**（Time Block 缩短 / 用户主动请求）：
- Agent 捕获 Time Block 变化上下文（哪天缩短、受影响的 Task）
- 只重新分配 **pending** 的 Task，已 completed 的不动
- 尊重 Goal 依赖顺序：已完成的前置 Goal 不影响后续 Goal 排期
- Agent 问用户：「你的学习时间缩短了。有以下选择：1→只调整今天 2→从今天起重新排 3→告诉我你的新安排」
- 生成调整 Proposal → 用户审批

**Proposal 升级**：
- `create_proposal`（study_plan 类型）新增：
  - `scheduled_date`：每个 Task 分配到哪一天
  - `estimated_minutes`：AI 估算耗时（内部参考，**不暴露给用户**）
  - Proposal Apply 时 `scheduled_date` 写入 Task 的 date 字段

**System Prompt 更新**：
- 新增「排期协议（Scheduling Protocol）」：
  - 排期前必须调用 `get_time_blocks` + `get_goal_dependencies`
  - 不锁定具体时间点，只分配到天
  - 时间估算仅内部参考，不暴露给用户
  - 尊重 Goal 依赖顺序
- 新增「重排协议（Rescheduling Protocol）」：
  - 检测到 Time Block 变化时的增量重排流程
  - 只动 pending Task，保留已完成进度

**验收标准**：
- [ ] Agent 在排期前自动获取 Time Block + Goal 依赖
- [ ] study_plan Proposal 中每个 Task 有 `scheduled_date`
- [ ] 排期尊重 Goal 依赖顺序（A 的 Task 排在 B 之前）
- [ ] 每天分配的 Must Task 总估时不超过该天 study Time Block 总时长
- [ ] 已有 Task 较多的日期被自动避开或减少分配
- [ ] AI 回复中不暴露时间估算数字
- [ ] Proposal Apply 后 Task 正确落到日历对应日期
- [ ] 增量重排：只动 pending Task，已完成的不变
- [ ] 重排时 Agent 提供 3 个选项让用户选择

### Step 4：L1 入驻流全链路（L1-1 ~ L1-5）

**目标**：新用户从注册到第一份学习计划落地日历，全链路自动化；排期引擎利用 Goal 依赖顺序 + Time Block 可用时长。

**工作内容**：
- 升级 Onboarding 组件：
  - Step 1（创建 Course）和 Step 2（上传材料）保持不变
  - Step 3 从「打开 AI 助手」改为「设置时间安排」：
    - 引导用户在**周视图网格上框选**设置学习和睡觉 Time Block（复用 Step 2 的 Time Block UI 组件）
    - 提供快速模板一键填充
    - 可跳过（AI 后续可再问）
  - 新增 Step 4：「让 AI 帮你规划」
    - 自动打开 Agent Panel
    - Agent 发出第一句话：「你在学什么？有什么学习目标或截止日期吗？」
    - 用户回复后进入参数收集对话流
- Agent L1 对话流（System Prompt 新增 L1 Protocol）：
  - 检测到新用户首次对话（onboarding_completed 刚完成）
  - 按顺序收集：学习目标 / 截止日期 → 已有 Time Block 确认或补充 → 颗粒度偏好 → 个性化要求
  - 每个问题一次只问一个，不堆积
  - 收集完毕后自动拆解目标 + 排期 + 生成 Proposal
  - **排期时自动读取 Goal 依赖顺序**（Step 3 排期引擎），确保前置 Goal 的 Task 优先分配
- Proposal 面板升级：
  - study_plan 类型 Proposal 显示日期分配（每个 Task 旁显示 scheduled_date）
  - Proposal 中如涉及多 Goal，按依赖顺序排列展示

### Step 5：设计宪法全面审计（DC-1 ~ DC-5）

**目标**：逐模块审计全产品（含 v1.3 新增功能），确保无一处违反设计宪法三条原则。

**工作内容**：

**审计维度**：
| 宪法条款 | 审计重点 | 违规示例 |
|---|---|---|
| 不替用户做决定 | AI 是否在未经 Proposal 机制的情况下直接创建/修改/删除数据 | 自动标记 Task 为逾期；重排提示自动执行而非让用户选择 |
| 不监控用户 | 产品是否追踪用时、判断精力、主动生成用户没要求的内容 | study_activity_log 的用途是否合规 |
| 不制造挫败感 | UI 是否有惩罚性元素、自动回顾失败、锁死时间 | 逾期 Task 红色高亮、streak 断裂提示、重叠 ℹ️ 提示措辞是否带负面暗示 |

**逐模块审计清单**：
- DailyBrief：检查 MWF 展示逻辑，是否存在自动排序/自动隐藏行为
- Statistics：检查是否暴露失败数据（未完成率、streak 断裂等），如有则改为被动展示
- Calendar：检查逾期 Task 的视觉处理，确保跳过零惩罚
- **Time Block 重叠处理**：叠加混合色 + ℹ️ 只做信息提示，不阻止保存、不弹警告弹窗
- **缩短重排提示**：确认「忽略」选项真正什么都不做，不暗示用户犯错
- **Goal 依赖 UI**：确认添加依赖是用户主动操作，Agent 不自动推断/创建依赖
- **增量重排**：确认 Agent 的 3 选项不含暗示性措辞（如「建议」某选项）
- Agent 全场景回归：遍历 System Prompt 所有指令段（含新增排期协议和重排协议），确认无违规行为
- study_activity_log 表审计：确认其用途是否合规，是否需要重新定位或移除
- Agent 工具 `generate_weekly_review` 和 `get_statistics_overview`：确认为被动调用
- Onboarding 和 L1 流程：确认每步可跳过（含 Time Block 框选步骤），跳过无惩罚
- 审计结果输出为 `docs/audits/design-constitution-v1.3.md`

**修复原则**：
- 违反「不替用户做决定」→ 加 Proposal 机制或需用户确认
- 违反「不监控用户」→ 移除或改为被动展示（用户点击才显示）
- 违反「不制造挫败感」→ 改为中性表述 / 移除惩罚性视觉

**验收标准**：
- [ ] 审计报告 `design-constitution-v1.3.md` 完成
- [ ] 所有发现的违规项已修复（含 v1.3 新增功能）
- [ ] Time Block 重叠 ℹ️ 提示措辞中性，不制造焦虑
- [ ] 缩短重排提示的「忽略」选项无任何副作用
- [ ] study_activity_log 用途明确记录
- [ ] Agent 全场景无违规
- [ ] Statistics 页面不主动展示失败数据

## 九、风险与阻塞项

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 排期算法复杂度高 | 多课程 + 截止日期 + Goal 依赖关系的排期是 NP 问题 | v1.3 只做贪心排期（截止日期近远 × 优先级 × 依赖拓扑序），不做最优解 |
| Goal 依赖环检测性能 | Goal 数量大时 DFS 可能慢 | v1.3 单用户 Goal 数预估 <100，DFS O(V+E) 完全可接受；未来量大再优化 |
| Time Block 跨午夜处理 | 边界情况多 | 统一转为分钟数（0-1440），跨午夜拆为两段 |
| L1 对话流过于复杂 | Agent 可能在对话中迷路 | System Prompt 用状态机风格定义 L1 步骤，每步有明确退出条件 |
| 设计宪法审计发现大量违规 | 修复工作量不可预估 | 审计先输出报告，按严重程度分批修复；P0 本版本修，P1 可延后 |
| Proposal 面板显示日期后 UI 过于拥挤 | 移动端体验下降（虽然目前不做移动端） | 日期用紧凑格式（M/D），hover 显示完整日期 |
| 老用户无 Time Block | Agent 排期降级 | 无 Time Block 时按截止日期均匀分配，不强制设置（宪法第二条） |
| Time Block 重叠的 Agent 理解 | 重叠区域 Agent 如何计算可用时长？ | 重叠区域不重复计算——取并集后统计 study 总分钟数 |
| DAG 兼容但 v1.3 只做线性链 | 用户可能期望多依赖 | UI 明确标注「前置目标」单选，后续版本解锁多依赖；DB 层已就绪 |
