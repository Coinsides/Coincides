# Coincides v1.4 — 版本规划

> **主题**：Time Block L1 补全 + 卡片数据模型升级
> **前置版本**：v1.3.2（8 项 Bug 修复 + Agent Section 支持）
> **目标**：补齐 v1.3 Time Block 的可视化缺口（刻度线、时间标签、右键操作），同时升级卡片数据模型（新增 Example / Condition 字段 + 详情放大），为 v1.5 复习体验升级打基础。

---

## 一、核心概念对齐

v1.3 完成了 Time Block 系统和工作流引擎，但日历视图缺少基础的可视化元素，影响可读性：
- **缺少时间刻度**：当前 Time Block 渲染在周视图上，但没有 24h 刻度线和横线网格，用户难以判断每个 Block 的精确时间位置
- **缺少时间标签**：Time Block 只显示标题，不显示起止时间
- **缺少编辑入口**：创建后无法直接编辑或删除，只能通过 Agent 操作

同时，卡片系统需要为 v1.5 的复习体验升级做数据准备：
- **Example 字段缺失**：所有卡片类型（Definition/Theorem/Formula/General）均无示例字段，影响学习效果
- **Condition 字段缺失**：Theorem 卡片缺少适用条件字段，而 v1.5 的"撕胶带"复习交互（CD-5）需要依赖此字段做分层遮挡
- **详情视图太小**：卡片详情弹窗空间不足，数学公式等长内容显示拥挤

---

## 二、设计宪法守则（v1.4 特别提醒）

v1.4 为纯 UI/数据模型升级，不涉及 AI 行为或用户行为追踪，设计宪法风险较低。仍需注意：

| 宪法条款 | v1.4 约束 |
|---|---|
| 不替用户做决定 | 右键菜单只提供编辑/删除选项，不自动执行任何操作 |
| 不监控用户 | 新字段（Example/Condition）为用户选填，不强制 |
| 不制造挫败感 | 现有卡片无需回填新字段，空值正常显示 |

## 四、Step 拆分

### Step 1：卡片数据模型升级（CD-2 + CD-3）

**目标**：为所有卡片类型新增 `example` 字段，为 Theorem 卡片新增 `condition` 字段。

**后端**：
1. 新建 migration：`cards` 表无需加列（content 是 JSON TEXT 字段），变更在 JSON 结构层面
2. 更新 `shared/types.ts` 中各模板类型的 content 接口：
   - `DefinitionContent` 增加 `example?: string`
   - `TheoremContent` 增加 `example?: string` + `condition?: string`
   - `FormulaContent` 增加 `example?: string`
   - `GeneralContent` 增加 `example?: string`
3. 更新 `executor.ts` 中 `normalizeCardContent()` 兼容新字段
4. 更新 Agent `create_card` 工具的 JSON Schema，让 AI 知道可以填写 example/condition

**前端**：
1. `CardModal.tsx`（创建/编辑弹窗）：各模板 tab 新增 Example 输入框；Theorem tab 新增 Condition 输入框
2. `CardTemplateContent.tsx`：各模板视图渲染新字段（DefinitionView/TheoremView/FormulaView/GeneralView）
3. `CardFlip.tsx`：翻转卡片背面显示新字段
4. `CardViewModal.tsx`：详情弹窗显示新字段

**验收标准**：
- [ ] 创建 Definition 卡片时可填写 Example，保存后详情/翻转/网格预览均正确显示
- [ ] 创建 Theorem 卡片时可填写 Example + Condition，保存后详情/翻转均正确显示
- [ ] 创建 Formula 卡片时可填写 Example
- [ ] 创建 General 卡片时可填写 Example
- [ ] 已有卡片不受影响（新字段为空时不显示空白区域）
- [ ] Agent 创建卡片时可填写 example/condition 字段
- [ ] KaTeX 数学渲染在新字段中正常工作

---

### Step 2：卡片详情视图放大（CD-1）

**目标**：CardViewModal 放大尺寸，使数学公式和新增字段有足够空间展示。

**变更**：
1. `CardViewModal.module.css`：增大弹窗 max-width/max-height，优化内部 padding 和字体大小
2. `CardFlip.module.css`：放大后的翻转卡片尺寸匹配
3. 响应式处理：小屏幕时保持合理比例

**验收标准**：
- [ ] 包含长数学公式的 Theorem 卡片在放大详情中完整显示，无溢出
- [ ] Example / Condition 新字段在放大视图中布局合理
- [ ] 弹窗不超出视口，有合理的 scroll 处理

---

### Step 3：Time Block 可视化补全（TB-L1a + TB-L1b）

**目标**：日历周视图添加 24h 刻度线和横线网格，Time Block 上显示时间标签。

**变更**：
1. `Calendar.tsx`：在周视图的时间轴区域渲染 0:00~23:00 刻度标签（左侧列）
2. `Calendar.module.css`：
   - 每小时一条横线（浅色虚线 or 细实线），形成网格
   - 半小时辅助线（可选，更浅）
   - 刻度标签样式（小字号、灰色）
3. Time Block 渲染：每个 Block 内部顶部或底部显示 `HH:MM - HH:MM` 时间标签
4. `Calendar.module.css`：时间标签样式（小字号，不抢主标题视觉权重）

**验收标准**：
- [ ] 周视图左侧显示 0:00~23:00 小时刻度，间距与 Time Block 像素对齐
- [ ] 每小时横线清晰可见但不喧宾夺主
- [ ] 每个 Time Block 显示起止时间，字号小于标题
- [ ] 现有 Task 事件在日历上的显示不受影响

---

### Step 4：Time Block 右键操作（TB-L1c）

**目标**：右键点击 Time Block 弹出上下文菜单，支持编辑和删除。

**变更**：
1. `Calendar.tsx`：Time Block 元素添加 `onContextMenu` 事件处理
2. 新建 `ContextMenu` 组件（或在 Calendar 内部实现）：
   - 菜单选项：「编辑」「删除」
   - 定位逻辑：跟随右键点击位置，不超出视口
   - 点击外部自动关闭
3. 编辑流程：弹出编辑弹窗 → 可修改 label / start_time / end_time / color → 调用 PUT API
4. 删除流程：确认弹窗 → 调用 DELETE API
5. 后端：确认 `PUT /api/time-blocks/:id` 和 `DELETE /api/time-blocks/:id` API 已存在，补充缺失的

**验收标准**：
- [ ] 右键 Time Block 弹出上下文菜单
- [ ] 编辑：可修改标签、起止时间、颜色
- [ ] 删除：确认后删除，日历即时刷新
- [ ] 左键行为不变（不影响已有交互）
- [ ] 菜单在视口边缘不溢出

## 五、技术风险

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 已有卡片 content JSON 无新字段 | 前端读到 undefined | 所有新字段使用 `?.` 可选链 + 空值不渲染 |
| Calendar 刻度与 Time Block 像素对齐 | 刻度和 Block 位置不匹配 | 复用现有 Block 定位算法的时间→像素转换函数 |
| 右键菜单在不同浏览器行为差异 | 菜单不弹出或位置偏移 | Electron 环境（Windows），浏览器差异风险低 |
