# v2.0 Engineering Spec - NoteBlock Foundation

**Created**: 2026-05-19  
**Branch**: `feat/v2.0-noteblock`  
**Status**: Step 1 implemented and verified
**Source Plan**: `docs/releases/v2.0-plan.md`

## 1. Purpose

This spec turns the v2.0 plan into concrete engineering work.

v2.0 must create a small but real NoteBlock foundation inside the current app. It should not implement the full Course Material Library, Material Reconciliation, AI note generation, Source Snapshot Viewer, or old Card/Deck migration.

The implementation should be additive, testable, and friendly to later v2.1+ work.

---

## 2. Current System Facts

Current backend facts:

- Express app mounts route modules in `server/src/index.ts`.
- SQLite schema is initialized from `server/src/db/schema.sql`.
- Incremental migrations live under `server/src/db/migrations/` and are tracked by `db_migrations`.
- Existing learning material is mainly represented by `card_decks`, `card_sections`, `cards`, `documents`, and `document_chunks`.
- Existing source fields on `cards` include `source_document_id`, `source_page`, and `source_excerpt`.
- Existing `documents` and `document_chunks` already provide a usable early source-reference target.
- Existing `proposals` table stores JSON payloads, but v2.0 does not need full typed proposal execution.

Current frontend facts:

- React/Vite client uses course, deck, card, document, task, and agent surfaces.
- Course pages are the right entry point for v2 Note/NoteBlock work.
- Existing KaTeX-friendly rendering patterns should be reused for formulas where possible.

## 4. Database Migration

Add migration `014_v2_note_foundation.ts` unless a newer migration number already exists at implementation time. If the migration list has advanced, use the next available number.

The migration is additive only.

### 4.1 operation_batches

Purpose: trace structural operations and prepare for future Proposal -> Review -> Apply flows.

Columns:

- `id TEXT PRIMARY KEY`
- `user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE`
- `course_id TEXT REFERENCES courses(id) ON DELETE CASCADE`
- `source_type TEXT NOT NULL DEFAULT 'manual'`
- `source_id TEXT`
- `label TEXT`
- `status TEXT NOT NULL DEFAULT 'applied'`
- `metadata TEXT NOT NULL DEFAULT '{}'`
- `created_at DATETIME DEFAULT CURRENT_TIMESTAMP`
- `applied_at DATETIME`
- `reverted_at DATETIME`

Indexes:

- `(user_id, course_id)`
- `(source_type, source_id)`

Allowed `source_type` values by convention:

- `manual`
- `proposal`
- `migration`
- `agent`

v2.0 may enforce these in validation code rather than SQLite constraints if that better matches the existing style.

### 4.2 notes

Purpose: course-rooted user-facing note containers.

Columns:

- `id TEXT PRIMARY KEY`
- `user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE`
- `course_id TEXT NOT NULL REFERENCES courses(id) ON DELETE CASCADE`
- `title TEXT NOT NULL`
- `description TEXT`
- `status TEXT NOT NULL DEFAULT 'active'`
- `source_kind TEXT NOT NULL DEFAULT 'manual'`
- `page_format TEXT NOT NULL DEFAULT 'flow'`
- `metadata TEXT NOT NULL DEFAULT '{}'`
- `operation_batch_id TEXT REFERENCES operation_batches(id) ON DELETE SET NULL`
- `created_at DATETIME DEFAULT CURRENT_TIMESTAMP`
- `updated_at DATETIME DEFAULT CURRENT_TIMESTAMP`
- `trashed_at DATETIME`

Indexes:

- `(user_id, course_id, status)`
- `(course_id, updated_at)`

Allowed `status` values by convention:

- `active`
- `archived`
- `trashed`

### 4.3 note_blocks

Purpose: course-rooted canonical learning blocks.

Columns:

- `id TEXT PRIMARY KEY`
- `user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE`
- `course_id TEXT NOT NULL REFERENCES courses(id) ON DELETE CASCADE`
- `block_type TEXT NOT NULL`
- `title TEXT`
- `content_json TEXT NOT NULL DEFAULT '{}'`
- `plain_text TEXT`
- `status TEXT NOT NULL DEFAULT 'active'`
- `source_kind TEXT NOT NULL DEFAULT 'manual'`
- `metadata TEXT NOT NULL DEFAULT '{}'`
- `operation_batch_id TEXT REFERENCES operation_batches(id) ON DELETE SET NULL`
- `created_at DATETIME DEFAULT CURRENT_TIMESTAMP`
- `updated_at DATETIME DEFAULT CURRENT_TIMESTAMP`
- `trashed_at DATETIME`

Indexes:

- `(user_id, course_id, status)`
- `(course_id, block_type)`
- `(course_id, updated_at)`

Committed v2.0 block types:

- `heading`
- `paragraph`
- `definition`
- `theorem`
- `proof`
- `formula`
- `example`
- `exercise`
- `answer`
- `sidenote`

Deferred block types:

- `diagram_crop`
- `image_crop`
- `table`
- `code`
- custom package-defined block types

### 4.4 note_block_placements

Purpose: connect notes to ordered blocks. A note is an ordered view over course-rooted NoteBlocks.

Columns:

- `id TEXT PRIMARY KEY`
- `note_id TEXT NOT NULL REFERENCES notes(id) ON DELETE CASCADE`
- `block_id TEXT NOT NULL REFERENCES note_blocks(id) ON DELETE CASCADE`
- `parent_placement_id TEXT REFERENCES note_block_placements(id) ON DELETE SET NULL`
- `order_index INTEGER NOT NULL`
- `display_mode TEXT NOT NULL DEFAULT 'default'`
- `display_overrides_json TEXT NOT NULL DEFAULT '{}'`
- `created_at DATETIME DEFAULT CURRENT_TIMESTAMP`
- `updated_at DATETIME DEFAULT CURRENT_TIMESTAMP`

Indexes:

- `(note_id, order_index)`
- `(block_id)`

Constraint:

- Prefer unique `(note_id, block_id)` in v2.0 unless implementation discovers a strong need to place the same block multiple times in one note.

### 4.5 note_block_sources

Purpose: early SourceReference support using existing uploaded documents and chunks.

Columns:

- `id TEXT PRIMARY KEY`
- `block_id TEXT NOT NULL REFERENCES note_blocks(id) ON DELETE CASCADE`
- `document_id TEXT REFERENCES documents(id) ON DELETE SET NULL`
- `document_chunk_id TEXT REFERENCES document_chunks(id) ON DELETE SET NULL`
- `source_page_start INTEGER`
- `source_page_end INTEGER`
- `source_excerpt TEXT`
- `reference_type TEXT NOT NULL DEFAULT 'page'`
- `confidence REAL`
- `metadata TEXT NOT NULL DEFAULT '{}'`
- `created_at DATETIME DEFAULT CURRENT_TIMESTAMP`

Indexes:

- `(block_id)`
- `(document_id)`
- `(document_chunk_id)`

- `updated_at DATETIME DEFAULT CURRENT_TIMESTAMP`
- `trashed_at DATETIME`

Indexes:

- `(user_id, course_id, type, status)`
- `(course_id, updated_at)`

v2.0 projection types:

- `organized_note`

Reserved future projection types:

- `review_set`
- `formula_sheet`
- `theorem_proof_list`
- `concept_focus`
- `study_scope_plan`

## 5. API Contract

All APIs require the existing authenticated user context. Every read/write must verify ownership through `user_id` and related `course_id`.

### 5.1 Notes

`GET /api/notes?course_id=...`

Returns active notes for a course by default. May accept `status` for `active`, `archived`, or `trashed`.

`POST /api/notes`

Request:

```json
{
  "course_id": "course-id",
  "title": "Limits and Continuity",
  "description": "optional",
  "page_format": "flow"
}
```

Response: created note.

`GET /api/notes/:id`

Returns the note and basic metadata.

`PUT /api/notes/:id`

Allows title, description, page_format, metadata, and status updates.

`DELETE /api/notes/:id`

Moves the note to `trashed` and sets `trashed_at`. It does not hard-delete in v2.0.

### 5.2 Note Blocks

`GET /api/notes/:id/blocks`

Returns ordered placements with block content and source references.

`POST /api/notes/:id/blocks`

Creates a course-rooted NoteBlock and adds a placement to the note.

Request:

```json
{
  "block_type": "definition",
  "title": "Gradient",
  "content_json": {
    "body": "The gradient points in the direction of steepest increase."
  },
  "plain_text": "The gradient points in the direction of steepest increase.",
  "source_references": [
    {
      "document_id": "document-id",
      "document_chunk_id": "chunk-id",
      "source_page_start": 12,
      "source_page_end": 12,
      "source_excerpt": "optional excerpt",
      "confidence": 0.9
    }
  ]
}
```

`PUT /api/note-blocks/:id`

Updates the block content, title, type, metadata, or status.

`DELETE /api/note-blocks/:id`

Moves the block to `trashed` and sets `trashed_at`. The placement should no longer appear in active note views.

`PUT /api/notes/:id/blocks/reorder`

Request:

```json
{
  "placements": [
    { "placement_id": "placement-a", "order_index": 0 },
    { "placement_id": "placement-b", "order_index": 1 }
  ]
}
```

Reorder must be scoped to one note.

### 5.3 Projections

`GET /api/projections?course_id=...&type=organized_note`

Returns projection snapshots for a course.

`POST /api/projections`

Creates a stable snapshot from a note or explicit block list.

Request:

```json
{
  "course_id": "course-id",
  "type": "organized_note",
  "title": "Week 1 Organized Note",
  "source_note_id": "note-id",
  "snapshot_json": {
    "blocks": []
  }
}
```

`GET /api/projections/:id`

Returns the projection snapshot. It should not silently re-render from live blocks.

## 6. Validation Rules

Add Zod validators for:

- note creation/update,
- note block creation/update,
- block reorder,
- source reference payload,
- projection creation.

Validation principles:

- `course_id` is required for creating notes and projections.
- `block_type` must be one of the committed v2.0 block types.
- `content_json` must be a JSON object.
- `plain_text` should be optional but recommended.
- `source_page_end` must be greater than or equal to `source_page_start` when both exist.
- metadata fields must be JSON objects, not arbitrary strings.
- status changes must use `active`, `archived`, or `trashed`.

## 7. Frontend Contract

### 7.1 Course Notes Entry

On the course detail page, add a Notes entry that lets the user:

- view notes for the current course,
- create a new note,
- open a note detail/editor view.

This should not remove or hide the existing Deck/Card path.

### 7.2 Note Detail / Editor

The note editor must support:

- note title editing,
- adding a block,
- editing block content,
- selecting a block type,
- reordering blocks,
- trashing a block,
- rendering formulas with existing KaTeX-friendly behavior,
- displaying simple source-reference text when present.

The editor does not need:

- canvas layout,
- A4 print layout,
- drag-select source regions,
- AI generation,
- proposal review UI,
- Source Snapshot Viewer.

### 7.3 UI Language

Use factual language.

Good:

- `Source reference`
- `This block is linked to page 12`
- `Move to trash`
- `Create note`

Avoid judgmental learning language.

Bad:

- `You are weak at this concept`
- `You must learn this first`
- `You are behind`

## 8. Migration Strategy

Implementation steps:

1. Add migration file.
2. Update `schema.sql` with the same new tables.
3. Run migration on an empty database.
4. Run migration on an existing v1 database.
5. Confirm existing Card/Deck data remains unchanged.

No automatic card-to-note migration in v2.0.

Future migration path reserved:

```text
card_decks / cards
  -> note_blocks
  -> review projections
  -> future v2 review system
```

---

## 9. OperationBatch Behavior

v2.0 should create operation batches for larger user-visible structural operations where practical:

- creating a note with initial blocks,
- bulk reorder,
- creating a projection snapshot.

Single-field edits may skip operation batches if adding them would create unnecessary complexity.

Future typed proposals should be able to point at `operation_batches`.

---

## 10. Projection Snapshot Behavior

Projection snapshots must be stable.

If a user creates a projection from a note, then later edits the note block, the existing projection should not change silently.

Minimum v2.0 approach:

- store enough rendered block data inside `snapshot_json`,
- store source block ids and timestamps inside `source_versions_json`,
- show projection content from snapshot data, not live block data.

Refreshing a projection can be a future explicit action.

---

## 11. Error Handling

Use existing API error conventions where possible.

Required error cases:

- course not found or not owned by user,
- note not found or not owned by user,
- block not found or not owned by user,
- placement does not belong to note,
- source document not owned by user or not in the same course,
- invalid block type,
- invalid status transition,
- invalid reorder payload.

## 14. Explicit Non-Goals

Do not implement these in v2.0:

- full AI note proposal,
- material upload reconciliation,
- source snapshot rendering,
- OCR routing,
- textbook-scale planning,
- rerank model,
- community package schemas,
- old Card/Deck deletion,
- local desktop packaging.

If any of these become necessary during implementation, pause and update the plan before coding further.
