# Coincides v1.7.1 — Agent 效率深度优化 实施计划

> **主题**：Agent Tool Efficiency 深度优化 — 并行执行 + 分场景 Playbook + Context 预注入  
> **前置版本**：v1.7（Task-Card 联动）  
> **目标**：大幅减少 Agent 工具调用轮次和总耗时。卡片生成从 7+ 轮降至 3–4 轮，学习计划从 6+ 轮降至 3–5 轮，目标拆解降至 2–3 轮，文档查找降至 1 轮。

---

## 一、问题背景

用户在测试中遇到 Agent 超时（120s），上传 2 份文档生成闪卡时请求中断。根因分析：

1. **SSE 超时不对齐**：`agent.ts` 的 SSE 请求超时为 120s，但 orchestrator 已设为 300s，导致 orchestrator 还在工作时 SSE 已断开
2. **Agent 工具调用低效**：顺序执行工具、缺少场景化指导、每次对话都要先 `list_decks` + `list_sections` 浪费 1–2 轮

---

## 二、设计宪法守则

| 宪法条款 | v1.7.1 约束 |
|---|---|
| 不替用户做决定 | 效率优化仅影响 Agent 内部执行路径，不改变用户交互流程；所有 Proposal 机制不变 |
| 不监控用户 | 预注入的 Deck/Section 数据仅用于减少工具调用，不用于主动推荐或分析 |
| 不制造挫败感 | 优化目标是减少等待时间，提升用户体验 |

## 五、Step 2：Orchestrator 并行工具执行

### 5.1 目标

当 Claude 在同一轮返回多个 `tool_use` 请求时，并行执行而非顺序等待。

### 5.2 文件变更清单

| 文件 | 操作 | 变更内容 |
|------|------|----------|
| `server/src/agent/orchestrator.ts` | **修改** | 工具执行从 `for...of` 顺序循环改为 `Promise.all` 并行执行 |

### 5.3 技术细节

**变更前**：
```typescript
for (const toolUse of toolUseBlocks) {
  const result = await executeTool(toolUse.name, toolUse.input, db, userId);
  // ...
}
```

**变更后**：
```typescript
const results = await Promise.all(
  toolUseBlocks.map(async (toolUse) => {
    const result = await executeTool(toolUse.name, toolUse.input, db, userId);
    return { toolUse, result };
  })
);
```

### 5.4 安全性分析

| 场景 | 风险 | 评估 |
|------|------|------|
| 多个 SQLite 写操作并行 | 无 | better-sqlite3 是同步驱动，所有写操作在 JS 主线程顺序执行，不存在并发写冲突 |
| 多个 Embedding API 调用并行 | 无 | 纯网络 I/O，并行反而更快 |
| 一个工具依赖另一个结果 | 无 | Claude 在同一轮返回的工具调用天然无依赖关系（有依赖时会分多轮） |

### 5.5 验收标准

- [x] 多工具同轮调用并行执行
- [x] 单工具调用行为不变
- [x] TypeScript 编译无新错误

## 六、Step 3：System Prompt Playbook 重写

### 6.1 目标

将 Tool Efficiency 从 4 条泛规则升级为完整的分场景 Playbook，让 Agent 在每种任务类型下都有明确的最优执行路径。

### 6.2 文件变更清单

| 文件 | 操作 | 变更内容 |
|------|------|----------|
| `server/src/agent/system-prompt.ts` | **修改** | 重写 `## Tool Efficiency` 为分场景 Playbook；更新「基于文档生成卡片」协议段；更新 Deck 选择 / Section 组织规则 |

### 6.3 Playbook 内容

#### 卡片生成（目标 3–4 轮）

| 轮次 | 操作 |
|------|------|
| Round 1 | 读取文档内容 `get_document_content`（如需要） |
| Round 2 | 批量创建卡片 `create_card` ×N（合并为一轮多调用） |
| Round 3 | 关联任务 `link_task_cards`（如适用） |
| Round 4 | 回复用户确认结果 |

#### 学习计划（目标 3–5 轮）

| 轮次 | 操作 |
|------|------|
| Round 1 | 查询现有目标 `list_goals` + 查询文档 `search_documents`（并行） |
| Round 2 | 创建目标或任务（如需要）|
| Round 3 | 批量创建任务 `create_task` ×N |
| Round 4 | 关联卡片 `link_task_cards`（如适用） |
| Round 5 | 回复用户确认 |

#### 目标拆解（目标 2–3 轮）

| 轮次 | 操作 |
|------|------|
| Round 1 | 创建子任务 `create_task` ×N（一轮完成所有子任务） |
| Round 2 | 回复用户确认 |

#### 文档查找（目标 1 轮）

| 轮次 | 操作 |
|------|------|
| Round 1 | `search_documents` 或 `get_document_content` + 回复 |

### 6.4 Anti-Patterns 清单

以下行为被明确禁止：

| ❌ 禁止行为 | ✅ 正确做法 |
|---|---|
| 先 `list_decks` 再 `list_sections` 分两轮 | 使用预注入的 Available Decks 上下文 |
| 每张卡片单独一轮创建 | 同一轮批量调用 `create_card` ×N |
| 创建卡片前先搜索确认不重复 | 除非用户明确要求，否则直接创建 |
| 空轮次（仅回复文字不调用工具） | 工具调用和回复合并在同一轮 |

### 6.5 验收标准

- [x] Playbook 覆盖 4 大场景（卡片、计划、目标、文档）
- [x] 每个场景有明确的目标轮次数
- [x] Anti-Patterns 清单完整
- [x] 关联协议段与 Playbook 一致
- [x] TypeScript 编译无新错误

## 七、Step 4：System Context 预注入

### 7.1 目标

在对话启动时预加载用户所有 Deck 和 Section 信息到 system prompt，消除 Agent 调用 `list_decks` / `list_sections` 的需要。

### 7.2 文件变更清单

| 文件 | 操作 | 变更内容 |
|------|------|----------|
| `server/src/agent/orchestrator.ts` | **修改** | 对话开始前查询所有 Deck + Section，传入 `buildSystemPrompt` |
| `server/src/agent/system-prompt.ts` | **修改** | `buildSystemPrompt` 签名新增可选 `decks` 参数；新增 `## Available Decks` 上下文区块 |

### 7.3 预注入数据格式

```
## Available Decks (pre-loaded — do NOT call list_decks / list_sections)

Deck: "AMATH 231" (id: abc123) — 45 cards — Course: AMATH 231
  Sections: "Chapter 1" (id: s1), "Chapter 2" (id: s2), "Chapter 3" (id: s3)

Deck: "CS 341" (id: def456) — 28 cards — Course: CS 341
  Sections: "Greedy" (id: s4), "DP" (id: s5)
```

### 7.4 数据查询

```typescript
// orchestrator.ts — 对话启动前
const decks = db.prepare(`
  SELECT d.id, d.name, d.course_id,
    (SELECT COUNT(*) FROM cards WHERE deck_id = d.id) as card_count,
    c.name as course_name
  FROM decks d
  LEFT JOIN courses c ON d.course_id = c.id
  WHERE d.user_id = ?
`).all(userId);

for (const deck of decks) {
  deck.sections = db.prepare(`
    SELECT id, name FROM sections WHERE deck_id = ? ORDER BY position
  `).all(deck.id);
}
```

### 7.5 效果预估

| 场景 | 优化前调用 | 优化后调用 | 节省轮次 |
|------|-----------|-----------|---------|
| 卡片生成（已知 Deck） | `list_decks` → `list_sections` → `create_card` ×N | `create_card` ×N | 2 轮 |
| 卡片生成（需选 Deck） | `list_decks` → 回复确认 → `list_sections` → `create_card` ×N | 回复确认 → `create_card` ×N | 1–2 轮 |
| 学习计划 | `list_decks` → `list_goals` → ... | `list_goals` → ... | 1 轮 |

### 7.6 验收标准

- [x] System prompt 包含 Available Decks 区块
- [x] Deck 信息含 ID、名称、卡片数、课程链接、Section 列表
- [x] Agent 不再需要调用 `list_decks` 获取基本信息
- [x] TypeScript 编译无新错误
