# Coincides v1.5.3 — 版本规划

> **主题**：Agent 智能规划升级：问卷收集 + 双模式排期 + 文档感知 + TB 工具
> **前置版本**：v1.5.2（Task-Block 关联完善）
> **目标**：Agent 制定学习计划前，通过结构化问卷主动收集用户偏好（排期模式、文档选择、已掌握内容、任务量上限）；支持 Time Block 模式与 Calendar Event 模式两种排期方式；Agent 具备 TB 创建/管理能力。

---

## 一、设计宪法守则

| 宪法条款 | v1.5.3 约束 |
|---|---|
| 不替用户做决定 | 问卷收集偏好后才生成计划；文档选择、知识点排除、排期模式均由用户决定 |
| 不监控用户 | 不根据用户历史自动推断偏好；不追踪用户在问卷上的停留时间 |
| 不制造挫败感 | 问卷所有项均可选/可跳过；不因用户跳过某项而降低计划质量 |

---

## 二、核心设计决策

### 2.1 问卷机制

**位置**：对话流内（非 Proposal 内）。Agent 发送一条特殊类型的消息，前端渲染为结构化问卷卡片。

**理由**：
- 问卷是计划生成的**输入**，Proposal 是计划生成的**输出**——不应混在一起
- 对话内问卷保持用户心智连贯：Agent 在了解我 → Agent 在帮我做计划 → 我审批

**触发时机**：每次 Agent 进入"制定学习计划"流程时，**必须**先发送问卷收集偏好，再生成 proposal。不能被动等用户提及。

**交互流**：
1. 用户："帮我根据笔记制定学习计划"
2. Agent 调用 `search_documents` 获取文档列表 → 调用 `collect_preferences` 工具发送问卷
3. 前端渲染问卷卡片，用户填写
4. 用户提交 → 作为结构化消息回传 Agent
5. Agent 根据偏好读取文档、提取知识点
6. （可选）Agent 发送第二轮问卷：知识点列表，让用户排除已掌握的
7. Agent 生成 proposal → 用户审批

### 2.2 双模式排期

| 模式 | 名称 | 描述 | 默认 |
|------|------|------|------|
| A | **Time Block 模式** | Agent 创建 TB，任务挂在 TB 下。用户描述时间范围，Agent 构建 TB 结构 + 分配任务 | 是 |
| B | **Calendar Event 模式** | 每个任务有明确的 start_time + end_time，直接显示在周视图上。用户在 Proposal 面板中手动设置时间 | 否 |

**模式选择方式（A+B 并行）**：
- **A 路径**：问卷中提供模式选择题（默认 Time Block）
- **B 路径**：全局设置面板可设置默认模式，Agent 读取用户 preference

### 2.3 文档读取限制

| 限制项 | 值 | 理由 |
|--------|-----|------|
| 问卷文档选择上限 | **3 个** | 3 × ~30 页 ≈ 90 页，留足 token 空间给计划+卡片生成 |
| 单次 Agent 会话文档总页数 | **≤100 页** | 200K context 需留空间给 system prompt + 对话历史 + proposal 生成 + 可能的卡片生成 |
| Agent 读取策略 | ≤50 页：全文读取；>50 页：semantic search + 前 5 chunk | 平衡完整性与 token 消耗 |
| `MAX_TOOL_ROUNDS` | **5 → 8** | 文档读取 + 问卷收集 + 计划生成需要更多轮次 |

## 四、Step 拆分

### Step 1：问卷基础设施（SP-01 + SP-02）

**目标**：建立 Agent ↔ 前端的问卷通信机制。

**后端变更**：
1. `server/src/agent/tools/definitions.ts` — 新增 `collect_preferences` 工具定义：
   ```
   name: 'collect_preferences'
   description: '向用户发送结构化问卷收集偏好。用于制定计划前收集排期模式、文档选择、任务量等。'
   parameters: {
     questions: [{
       id: string,           // 问题标识符
       type: 'single_choice' | 'multi_choice' | 'number_input' | 'document_select',
       label: string,         // 问题文本
       options?: [{ value: string, label: string, description?: string }],
       default_value?: string | string[],
       required?: boolean,
       max_select?: number,   // multi_choice / document_select 的选择上限
       documents?: [{ id: string, filename: string, page_count: number, summary: string }]  // document_select 类型专用
     }]
   }
   ```
2. `server/src/agent/tools/executor.ts` — `collect_preferences` handler：
   - 返回一个特殊标记（如 `{ type: 'preference_collection', questions: [...] }`）
   - Orchestrator 识别此标记后，将问卷数据通过 stream 发送给前端
3. `server/src/agent/orchestrator.ts` — 新增 stream chunk type `preference_form`：
   - 当 tool result 包含 `type: 'preference_collection'` 时，yield 一个 `{ type: 'preference_form', data: questions }` chunk
   - Agent 循环暂停，等待用户回复（实际上是正常结束当前轮，用户回复后新一轮开始）

**前端变更**：
1. `client/src/components/AgentPanel/PreferenceForm.tsx` — 新组件：
   - 渲染问卷卡片：单选（radio）、多选（checkbox）、数字输入、文档选择列表
   - 文档选择：显示文件名 + 页数 + 摘要，最多选 `max_select` 个
   - 底部"提交"按钮
   - 提交后将选择结果序列化为结构化文本消息发送给 Agent
2. `client/src/components/AgentPanel/AgentPanel.tsx` — 消息渲染逻辑：
   - 识别 `preference_form` 类型的消息
   - 渲染 `<PreferenceForm>` 组件
   - 提交后替换为只读状态（显示用户已选择的内容）
3. `client/src/components/AgentPanel/AgentPanel.module.css` — 问卷卡片样式

### Step 2：Agent TB 工具扩展（SP-03 + SP-06）

**目标**：Agent 获得 Time Block 创建/管理能力。

**变更**：
1. `server/src/agent/tools/definitions.ts` — 新增 3 个工具：
   - `create_time_blocks`：批量创建 TB（复用 `POST /api/time-blocks` 逻辑）
     ```
     parameters: {
       blocks: [{
         label: string,
         type: 'study' | 'rest' | 'meal' | 'exercise' | 'custom',
         day_of_week: number (0-6),
         start_time: string (HH:MM),
         end_time: string (HH:MM),
         color?: string
       }]
     }
     ```
   - `update_time_block`：更新单个 TB（label / start_time / end_time / type）
   - `delete_time_block`：删除单个 TB
2. `server/src/agent/tools/executor.ts` — 实现 3 个 handler：
   - `create_time_blocks`：直接操作 DB（同 `POST /api/time-blocks` 逻辑），返回创建的 block ID 列表
   - `update_time_block`：同 `PUT /api/time-blocks/:id` 逻辑
   - `delete_time_block`：同 `DELETE /api/time-blocks/:id` 逻辑
3. `server/src/agent/orchestrator.ts` — `MAX_TOOL_ROUNDS` 从 5 → 8

### Step 3：System Prompt 升级（SP-04 + SP-07）

**目标**：Agent 知道何时发问卷、如何使用双模式排期、如何读取文档。

**变更**：
1. `server/src/agent/system-prompt.ts` — 新增「计划前偏好收集协议」：
   ```
   ## Pre-Planning Preference Collection（计划前偏好收集）
   当用户要求制定学习计划时，你必须在生成 proposal 之前收集偏好。
   
   步骤：
   1. 调用 search_documents 获取该 course 下的文档列表
   2. 调用 collect_preferences 发送问卷，包含：
      - 排期模式：Time Block 模式（默认）/ Calendar Event 模式
      - 文档选择：列出 course 下所有已解析文档（最多选 3 个，总页数 ≤100）
      - 每日任务量上限：3 / 5 / 7 / AI 决定
      - 计划粒度：精细（小任务多）/ 适中 / 粗略（大任务少）
      - 开放补充：文本框，用户可写额外要求
   3. 用户提交后，根据选择读取文档
   4. 读取后提取知识点清单，（可选）发送第二轮问卷让用户排除已掌握的
   5. 根据最终参数生成 proposal
   
   文档读取规则：
   - 总页数 ≤50：用 get_document_content 全文读取
   - 总页数 >50：用 search_documents semantic search 抽取关键段落 + 读取前 5 chunk
   - 绝对不超过 100 页总量
   ```
2. 新增「双模式排期协议」：
   ```
   ## Dual Scheduling Mode（双模式排期）
   
   ### Time Block 模式（默认）
   - 用户描述时间范围（如"明天8点到18点学习"）→ 调用 create_time_blocks 创建 TB
   - 任务挂到 TB 下（create_proposal 的 item 中设 time_block_id）
   - 任务只有 scheduled_date，不设 start_time/end_time
   - 遵守现有排期协议：不锁死时间段
   
   ### Calendar Event 模式
   - 任务有明确的 start_time + end_time
   - Agent 在 create_proposal 的 item 中设置 start_time / end_time
   - 但因为口述 8 个任务的时间不现实，Agent 只设 scheduled_date + 建议时段
   - 用户在 Proposal 面板中手动调整每个任务的 start_time / end_time
   
   模式选择来自问卷结果或用户 settings 中的 default_scheduling_mode。
   ```
3. 更新「MWF Study Plan Creation Flow」引用新协议
4. 更新「L1 Protocol」：新用户首次制定计划也走问卷流程

### Step 4：Proposal 时间编辑 UI（SP-05）

**目标**：Calendar Event 模式下，用户可在 Proposal 面板中为每个任务设置 start_time / end_time。

**变更**：
1. `client/src/components/AgentPanel/ProposalList.tsx` — 每个 item 新增时间编辑：
   - 如果 proposal type 为 `study_plan` 且 scheduling_mode 为 `calendar_event`（或 item 已有 start_time/end_time）：
     - 显示 start_time / end_time 字段
     - 点击可弹出时间选择器（HH:MM 格式）
     - 编辑后调用 `updateProposal` 保存
   - 如果是 Time Block 模式：显示关联的 TB 名称（只读）
2. `client/src/components/AgentPanel/TimePickerInline.tsx` — 内联时间选择器组件
3. `client/src/components/AgentPanel/ProposalList.module.css` — 时间编辑样式

## 五、关键依赖与风险

| 风险 | 影响 | 缓解方案 |
|------|------|----------|
| 问卷机制是全新的 stream chunk 类型，可能影响现有消息渲染 | 中 | 前端做好消息类型 fallback，未知类型渲染为纯文本 |
| Agent 用 `collect_preferences` 后需要"等待"用户回复——这不是真正的暂停，而是正常的对话轮转 | 低 | 无需改 orchestrator 流程，问卷提交后自然触发新一轮 |
| 大文档（80-100 页）读取时 Agent 工具轮次消耗多 | 中 | `MAX_TOOL_ROUNDS=8` + 智能读取策略（semantic search 优先） |
| `create_time_blocks` 创建的 TB 是 template（每周重复），用户可能只想在特定日期用 | 中 | 后续版本考虑支持"一次性 TB"（override 机制），v1.5.3 先用 template |
| 第二轮问卷（知识点排除）在大文档场景下知识点可能很多 | 中 | 设知识点列表上限（如 20 个），超出时合并为更粗的主题 |
