# v2.0 Plan - NoteBlock Foundation

**Created**: 2026-05-16  
**Updated**: 2026-05-19  
**Branch**: `feat/v2.0-noteblock`  
**Status**: ready for implementation / pre-implementation

---

## 1. Goal

Build the minimum durable foundation for the v2 NoteBlock system.

v2.0 introduces `NoteBlock` as the future canonical learning-material unit, keeps existing v1 Card/Deck flows usable, and prepares the app for v2.1 AI Note Proposal and Course Material Library work.

v2.0 is not a full AI note-generation release. It is the structural base that lets future releases organize, reference, project, and migrate learning material without being trapped by the old Card/Deck model.

Core outcome:

```text
Course
  -> NoteBlock Library foundation
  -> Notes / organized views
  -> Projection-compatible snapshots
  -> SourceReference placeholders
  -> status/trash-safe mutation model
```

---

## 2. Product Position

Coincides is moving toward a Personal Learning Material OS / learning steward.

v2.0 supports that direction by creating a stable learning-material substrate. It does not try to teach the user, judge the user, or decide what the user must learn. It gives future AI workflows a safer place to organize user-owned learning materials.

User-facing principle:

```text
The user owns the learning decision.
The app organizes material and proposes changes.
The user reviews and applies changes.
```

---

## 3. Version Boundary

v2.0 is locked to foundation work.

v2.0 must include:

- a real storage model for course-rooted notes and note blocks,
- stable CRUD APIs for notes and note blocks,
- a minimal UI surface to create, view, edit, reorder, and trash notes / blocks,
- source-reference fields that can point to existing uploaded documents and page ranges,
- a projection snapshot foundation, even if only the simplest organized-note projection is exposed,
- operation-batch tracing for future Proposal -> Review -> Apply compatibility,
- status/trash behavior for new v2 entities,
- additive migrations that do not rewrite or delete v1 Card/Deck data,
- tests and documentation updates that prove v1 data remains safe.

v2.0 must not include:

- full OCR pipeline,
- full AI note layout generation,
- full Course Material Library UI,
- Material Reconciliation,
- canonical source deduplication,
- Adaptive Layered Summaries,
- Source Snapshot Viewer / PDF Reader,
- bbox or region-selection UI,
- infinite canvas,
- Package Studio,
- community package marketplace,
- textbook-scale ingestion,
- rerank model integration,
- cross-course concept linking,
- desktop packaging,
- hosted/cloud service migration,
- forced migration from old cards into new NoteBlocks.

## 4. Implementation Scope

### 4.1 Backend

Add new v2 storage and API layers while leaving existing Card/Deck routes intact.

Committed backend work:

- Add additive database migration for v2 foundation tables.
- Add route modules for notes, note blocks, and projections.
- Add validators for all new request payloads.
- Add server-side ownership checks through the existing authenticated user model.
- Add status/trash behavior for new v2 entities.
- Add simple source-reference support against existing `documents` and `document_chunks`.
- Add projection snapshot records that remain stable after creation.
- Add operation-batch records for user-visible structural operations.

Existing routes that must remain usable:

- `/api/courses`
- `/api/decks`
- `/api/cards`
- `/api/documents`
- `/api/proposals`
- `/api/review`

### 4.2 Frontend

Add the smallest product surface that proves the foundation.

Committed frontend work:

- Add a Notes entry point inside the Course detail experience.
- Add a note list for a course.
- Add a note detail/editor view.
- Support creating and renaming notes.
- Support adding, editing, deleting/trashing, and reordering note blocks.
- Render the initial block types with existing text and KaTeX-friendly patterns.
- Show simple source-reference metadata when a block has attached references.
- Keep old Card/Deck navigation accessible.

v2.0 UI should feel like a clean foundation, not a final canvas product.

### 4.3 Documentation

Keep the v2 document stack aligned:

- `docs/Coincides-Roadmap.md` remains the formal version direction.
- `docs/brainstorm/v2.x-brainstorm.md` remains the idea pool.
- `docs/releases/v2.0-plan.md` locks the current version boundary.
- `docs/releases/v2.0-engineering-spec.md` defines concrete implementation choices.
- `docs/ARCHITECTURE.md` and `docs/DATA_MODEL.md` stay aligned with actual scope after implementation.

## 5. Data Model Decisions

v2.0 real entities:

- `Note`
- `NoteBlock`
- `NoteBlockPlacement`
- `NoteBlockSource`
- `Projection`
- `ProjectionSnapshot` as stored snapshot data inside the projection record
- `OperationBatch`
- status/trash metadata for new v2 entities

v2.0 does not create full tables for:

- `CourseMaterialLibrary`
- `SourceMaterial`
- `SourceSnapshot`
- `SourceFragment`
- `MaterialSegment`
- `MaterialSegmentSummary`
- `MergedSourceEvidence`
- `AdaptiveSummary`
- `SummaryDependency`
- `Concept`
- `ConceptMention`
- `LearningInboxItem`
- Package Studio schemas

These remain conceptual or future-version entities until promoted by later version plans.

Important modeling choice:

```text
NoteBlock is course-rooted.
Note is an ordered view over NoteBlocks.
NoteBlockPlacement connects a Note to a NoteBlock.
```

This keeps v2.0 compatible with the future Course Material Library without forcing a complete material library in this release.

## 6. API / Contracts

v2.0 implements a small explicit API surface.

Required route groups:

- `GET /api/notes?course_id=...`
- `POST /api/notes`
- `GET /api/notes/:id`
- `PUT /api/notes/:id`
- `DELETE /api/notes/:id`
- `GET /api/notes/:id/blocks`
- `POST /api/notes/:id/blocks`
- `PUT /api/notes/:id/blocks/reorder`
- `PUT /api/note-blocks/:id`
- `DELETE /api/note-blocks/:id`
- `GET /api/projections?course_id=...`
- `POST /api/projections`
- `GET /api/projections/:id`

Delete behavior:

- `DELETE` means move to trash for v2 user-owned entities.
- Permanent delete is not required in v2.0.

Contract principles:

- All write APIs must be scoped to the authenticated user.
- All NoteBlock writes must be scoped to a course.
- Reordering must be explicit and stable.
- Source references are optional in v2.0 but must have a stable shape.
- Projection records must store snapshots rather than silently changing when underlying blocks change.

## 7. Agent Tool Contract

v2.0 does not implement full AI Note Proposal.

v2.0 prepares for it by making all structural operations compatible with future typed proposals:

- create note,
- create block,
- update block,
- reorder blocks,
- attach source reference,
- create projection snapshot,
- trash note or block.

Future proposal payloads should be able to express these operations without changing the v2.0 storage model.

Rule:

```text
No agent should apply large structural note changes without a proposal record and user approval.
```

v2.1 will decide the first real AI Note Proposal payloads and apply flow.

---

## 8. Migration

v2.0 migration is additive.

Migration rules:

- Do not rewrite existing Card/Deck tables.
- Do not auto-convert old cards into NoteBlocks in v2.0.
- Do not delete old card/deck review data.
- Preserve FSRS state where it already exists.
- Add new tables through the existing migration system.
- Update `schema.sql` so new installs and migrated installs converge.
- Test empty database and existing v1-style database paths.

Relationship to v1 Card/Deck:

- v1 Card/Deck remains a legacy/current feature.
- v1 Card/Deck is treated as an ancestor of NoteBlock, not a peer final model.
- Later versions may migrate cards into NoteBlocks and review projections.
- v2.0 should avoid decisions that make that migration harder.

## 9. Test Matrix

Codex-side checks:

- database migration runs on an empty database,
- database migration runs on an existing v1-style database,
- server TypeScript build passes,
- client TypeScript/Vite build passes if UI changes,
- new API route smoke checks pass,
- reorder and trash behaviors are verified,
- projection snapshot does not silently mutate after source block edit,
- existing card/deck API smoke checks still pass,
- documentation links remain valid,
- secret scan finds no real provider keys.

Henry-side checks:

- Windows local run works,
- existing courses/cards/decks still feel accessible,
- Notes entry point is understandable from a course page,
- block editing feels like a believable foundation for future organized notes,
- source-reference language is factual and not overconfident,
- v2.0 feels like it can support v2.1 AI Note Proposal without contradicting the product direction.

---

## 10. Acceptance Criteria

v2.0 is complete when:

- a user can create a course-rooted note,
- a user can create, edit, reorder, and trash note blocks,
- note blocks are stored as course-rooted learning material rather than old card records,
- a note can be represented as an ordered view over blocks,
- a block can optionally point to existing source document/page evidence,
- a projection snapshot can be created and remains stable,
- existing v1 Card/Deck data remains available and unchanged,
- migrations are additive and verified,
- API contracts are clear enough for v2.1 AI Note Proposal,
- docs agree on v2.0 scope and do not imply full Course Material Library is already implemented.
