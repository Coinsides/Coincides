# v2.0 Plan — NoteBlock Foundation

**Created**: 2026-05-16  
**Updated**: 2026-05-17  
**Branch**: `feat/v2.0-noteblock`  
**Status**: draft / pre-implementation

---

## 1. Goal

Build the minimum durable foundation for v2 NoteBlock Library.

v2.0 should make it possible to introduce NoteBlock as the future canonical learning-material unit while keeping existing v1 Card/Deck flows understandable and usable during migration.

The purpose of v2.0 is foundation, not full AI note generation, full Course Material Library, or full material reconciliation.

---

## 2. Scope

v2.0 may include:

- conceptual and storage model for NoteBlock Library,
- Course-rooted learning content model,
- minimal Note / NoteBlock / Projection vocabulary,
- early ProjectionSnapshot compatibility decisions,
- compatibility strategy for existing Card/Deck data,
- API contracts for creating, reading, updating, and grouping NoteBlocks,
- source-traceability fields or placeholders needed by later SourceFragment and SourceReference work,
- proposal contract compatible with future TypedProposal operations,
- status/trash fields where they naturally fit without over-expanding the version,
- summary dependency/stale concepts in docs and API design where relevant,
- migration plan from v1 Card/Deck toward NoteBlock and Review Projection,
- tests for schema, migration, API, and basic UI smoke paths,
- documentation updates that keep roadmap, architecture, data model, and workflow aligned.

---

## 3. Out Of Scope

v2.0 should not try to finish:

- full OCR pipeline,
- full AI note layout generation,
- full Course Material Library UI,
- full Material Reconciliation,
- adaptive summary generation engine,
- Source Snapshot Viewer / PDF Reader,
- region select or bbox highlight UI,
- infinite canvas,
- Package Studio,
- community package marketplace,
- textbook-scale ingestion,
- rerank model integration,
- cross-course concept linking,
- desktop packaging,
- hosted/cloud service migration,
- replacement of all old Card/Deck UI in one step.

These can be planned for v2.1+ after the foundation is stable.

---

## 4. API / Contracts

Before implementation, define the minimum API surface for:

- NoteBlock CRUD,
- Note or collection membership,
- projection references,
- source reference attachment or placeholder,
- proposal payloads that create/update/reorder NoteBlocks,
- migration/read compatibility for existing cards,
- status/trash behavior if implemented,
- version/snapshot behavior where projections touch live NoteBlocks.

API design standard:

- stable enough for frontend and agent tools,
- small enough to avoid premature framework building,
- explicit about side effects,
- compatible with Proposal -> Review -> Apply,
- not blocking future SourceFragment, MaterialSegment, TypedProposal, or ProjectionSnapshot work.

---

## 5. Data Model

Target conceptual entities for v2.0-level thinking:

- `Note`,
- `NoteBlock`,
- `Projection`,
- `ProjectionSnapshot`,
- `ReviewProjection`,
- `SourceReference`,
- `OperationBatch`,
- status/trash metadata where needed.

Potential future entities that may be designed but not fully implemented in v2.0:

- `Workspace`,
- `CourseMaterialLibrary`,
- `SourceMaterial`,
- `SourceSnapshot`,
- `SourceFragment`,
- `MaterialSegment`,
- `MaterialSegmentSummary`,
- `MergedSourceEvidence`,
- `AdaptiveSummary`,
- `SummaryDependency`,
- `TypedProposal`,
- `Concept`,
- `ConceptMention`,
- `LearningInboxItem`,
- package/style schemas.

v2.0 must explicitly decide whether any of these are real tables, placeholders, metadata fields, or deferred design notes.

---

## 6. Agent Tool Contract

v2.0 should define, even if not fully expose, how agent operations will interact with NoteBlocks.

Questions to answer:

- What tool creates a NoteBlock proposal?
- What tool reads existing NoteBlocks?
- What tool links NoteBlocks to source material or source placeholders?
- What tool creates or updates a projection?
- What fields must be returned for user review?
- What operation is forbidden without explicit user approval?
- How can the contract later become a typed proposal without breaking v2.0 work?

No agent tool should directly apply large structural changes without a proposal record.

---

## 7. Migration

v2.0 migration must protect existing v1 data.

Initial migration stance:

- v1 Card/Deck remains readable.
- Card/Deck is treated as a migration source, not the final model.
- Old cards may later become NoteBlocks or Review Projection items.
- Old decks may later become projection groups or note collections.
- FSRS state should be preserved when the same review unit survives.
- Destructive cleanup is deferred until the new model is proven.
- Deletion should prefer status/trash semantics over hard delete where user material is involved.

Migration work must be tested against both empty and existing databases.

---

## 8. Test Matrix

Codex-side checks:

- schema/migration runs on empty database,
- schema/migration runs on existing v1-style database,
- TypeScript compile/build if code changes,
- API route smoke tests for new contracts,
- proposal payload validation,
- browser smoke for changed UI,
- documentation consistency check,
- secret scan for committed docs.

Henry-side checks:

- Windows local run,
- existing v1 cards/decks still feel accessible,
- new NoteBlock concepts make sense from a learning-product perspective,
- AI/proposal language remains factual and non-judgmental,
- foundation feels like it can support Course Material Library, SourceReference, ProjectionSnapshot, and TypedProposal later.

---

## 9. Acceptance Criteria

v2.0 can be considered complete when:

- NoteBlock Foundation is documented and implemented at the agreed minimum scope,
- existing v1 learning material is not lost,
- APIs and agent contracts are clear enough for v2.1 AI Note Proposal,
- migration strategy is tested and reversible enough for real local data,
- roadmap, architecture, data model, workflow, and changelog agree,
- Henry can explain how Card/Deck evolves into NoteBlock Library without contradiction,
- the implementation does not block later Course Material Library, Source Snapshot Viewer, Material Reconciliation, Adaptive Summaries, or Typed Proposal work.

---

## 10. Open Questions

- Should v2.0 implement NoteBlock UI immediately, or start with storage/API and a minimal internal viewer?
- How much of SourceReference should be real in v2.0 versus reserved for v2.1?
- Should existing cards be copied into NoteBlocks early, or left as legacy records until the projection system is ready?
- What is the smallest useful review projection that proves the new model?
- Which current card/deck APIs should be wrapped versus replaced?
- What projection snapshot behavior is the minimum safe version for v2.0?
