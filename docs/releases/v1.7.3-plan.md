# Coincides v1.7.3 — Time Block 模板化：模板 + 实例化 + 独立编辑

> **主题**：Time Block 架构重构 — 从"周模板自动重复"改为"模板蓝图 + 显式实例化 + 实例独立"  
> **前置版本**：v1.7.2（学习计划流程完善）  
> **目标**：解决"改一天 Time Block 所有同星期的天都变了"的核心问题。用户可以创建多套周模板，选择日期范围应用，应用后每天的 Time Block 互相独立。

---

## 一、问题背景

当前 Time Block 使用 `day_of_week`（0-6）做周模板，意味着：

1. **改一个影响全部**：用户在周六加了一个 Time Block → 每周六都自动出现同样的 Block
2. **无法按天定制**：虽然有 `time_block_overrides` 表，但用户完全不知道这个机制的存在
3. **模板和实例混淆**：用户以为自己在编辑"这一天"的 Time Block，实际上改的是全局模板

### 根因

`time_blocks` 表同时承担了"模板定义"和"日历展示"两个职责，没有将蓝图（模板）与实际展示（实例）分离。

---

## 二、设计宪法守则

| 宪法条款 | v1.7.3 约束 |
|---|---|
| 不替用户做决定 | 模板应用到日期范围必须用户手动操作；覆盖已有 Time Block 需要用户确认 |
| 不监控用户 | 不自动检测用户的 Time Block 使用情况并建议优化 |
| 不制造挫败感 | 模板为空时不警告；删除某天的 Time Block 不弹确认（非破坏性操作，可再次应用模板恢复） |

---

## 三、核心设计决策（已确认）

### 3.1 数据模型：模板 + 实例分离

```
time_block_templates（周模板蓝图，无具体日期）
├── id, user_id, template_set_id
├── label, type, day_of_week (0-6), start_time, end_time, color

time_block_template_sets（模板集，如"正常周"、"考试周"）
├── id, user_id, name, created_at, updated_at

time_blocks（实例，绑定具体日期）—— 复用现有表，ALTER 结构
├── id, user_id, template_id (nullable, 溯源用)
├── label, type, date (YYYY-MM-DD), start_time, end_time, color
```

- **模板集**（template_sets）：支持多套模板，如"正常周"和"考试周"
- **模板条目**（templates）：属于某个模板集，定义周一到周日的时间段
- **实例**（time_blocks）：绑定到具体日期，独立存在，改一个不影响其他
- `time_block_overrides` 表废弃删除（实例本身就是按日期的，不需要 override）

### 3.2 模板编辑器

- **入口**：周视图右上角"模板"按钮
- **UI**：Modal 内展示纯周视图（周一到周日，无具体日期），可增删改 Time Block 条目
- **模板集切换**：Modal 顶部 Tab 或下拉菜单切换不同模板集，可新建/重命名/删除模板集
- **应用**：编辑完点"应用到日期范围" → MonthCalendar 选日期 → 确认

### 3.3 应用逻辑

1. 用户选中日期范围（如 3月22日 - 4月5日）
2. 系统按模板中每个 `day_of_week`，为对应日期生成 `time_blocks` 实例
3. **冲突处理**：如果某天已有 Time Block → 弹出确认"以下日期已有 Time Block，是否覆盖？" → 用户确认后覆盖（先删旧实例，再插新实例）
4. 生成的实例 `template_id` 记录来源，但此后独立

### 3.4 日历上编辑

- 点击/拖拽某天的 Time Block → 直接编辑/删除那一天的**实例**
- 不会影响模板，也不会影响其他天

### 3.5 迁移策略

- **清空** `time_blocks` 和 `time_block_overrides` 表数据
- **删除** `time_block_overrides` 表
- **ALTER** `time_blocks` 表：去掉 `day_of_week`，加 `date TEXT NOT NULL`、`template_id TEXT`
- 新建 `time_block_template_sets` 和 `time_block_templates` 表
- 同步清空 `tasks.time_block_id` 引用（因为 time_block 实例 ID 全部重新生成）

### 5.3 新表结构

```sql
-- 模板集
CREATE TABLE IF NOT EXISTS time_block_template_sets (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 模板条目
CREATE TABLE IF NOT EXISTS time_block_templates (
  id TEXT PRIMARY KEY,
  template_set_id TEXT NOT NULL REFERENCES time_block_template_sets(id) ON DELETE CASCADE,
  user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  label TEXT NOT NULL,
  type TEXT NOT NULL DEFAULT 'study',
  day_of_week INTEGER NOT NULL,   -- 0=Sun, 6=Sat
  start_time TEXT NOT NULL,
  end_time TEXT NOT NULL,
  color TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 实例（原 time_blocks 表 ALTER 后）
```

### 5.4 API 设计

#### 模板集
| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/time-block-templates/sets` | GET | 获取所有模板集 |
| `/api/time-block-templates/sets` | POST | 创建模板集 `{ name }` |
| `/api/time-block-templates/sets/:id` | PUT | 重命名模板集 `{ name }` |
| `/api/time-block-templates/sets/:id` | DELETE | 删除模板集（CASCADE 删除其下模板条目） |

#### 模板条目
| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/time-block-templates/sets/:setId/items` | GET | 获取某模板集的所有条目 |
| `/api/time-block-templates/sets/:setId/items` | POST | 批量创建/替换模板条目 `{ items: [...] }` |

#### 实例化
| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/time-block-templates/sets/:setId/apply` | POST | 应用模板到日期范围 `{ dates: string[], overwrite: boolean }` |

#### 实例（复用现有路由路径）
| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/time-blocks?date=YYYY-MM-DD` | GET | 获取某天的 Time Block 实例 |
| `/api/time-blocks?from=&to=` | GET | 获取日期范围内的实例 |
| `/api/time-blocks/:id` | PUT | 编辑单个实例（只影响这一天） |
| `/api/time-blocks/:id` | DELETE | 删除单个实例 |

### 5.5 实例化逻辑

```typescript
// POST /api/time-block-templates/sets/:setId/apply
// Body: { dates: ["2026-03-22", "2026-03-23", ...], overwrite: boolean }

for (const date of dates) {
  const dayOfWeek = new Date(date).getDay();
  const templateItems = templates.filter(t => t.day_of_week === dayOfWeek);
  
  if (overwrite) {
    // 删除该日期已有的所有 time_blocks
    db.prepare('DELETE FROM time_blocks WHERE user_id = ? AND date = ?').run(userId, date);
  } else {
    // 检查是否已有，有则跳过
    const existing = db.prepare('SELECT id FROM time_blocks WHERE user_id = ? AND date = ?').get(userId, date);
    if (existing) continue;
  }
  
  for (const tmpl of templateItems) {
    db.prepare('INSERT INTO time_blocks (...) VALUES (...)').run(
      uuidv4(), userId, tmpl.id, tmpl.label, tmpl.type, 
      date, tmpl.start_time, tmpl.end_time, tmpl.color, now, now
    );
  }
}
```

### 7.3 日历渲染变化

**之前**：
```
fetchWeek(date) → server 查 day_of_week 模板 + overrides → 合并为 resolved blocks → 渲染
```

**之后**：
```
fetchTimeBlocks(from, to) → server 查 date-based 实例 → 直接渲染
```

逻辑大幅简化：不再需要 resolve 步骤。

## 十、关键依赖与风险

| 风险 | 影响 | 缓解方案 |
|------|------|----------|
| 数据清空导致用户丢失已有 Time Block | 中 | 产品仍在早期，用户确认可接受清空重来 |
| time_blocks 表结构变化导致大量代码适配 | 高 | 分 Step 推进，每 Step 编译检查 |
| 多套模板增加 UI 复杂度 | 中 | 默认创建"默认模板"集，用户不主动创建多套时体验不变 |
| Agent 缺口检测从 day_of_week 改为 date | 低 | 逻辑更简单（直接查实例有无），不易出错 |
| Calendar 渲染大改 | 高 | resolved block 逻辑删除后反而更简单，但需要仔细测试翻周边界 |

## 十一、与现有功能的关系

```
模板编辑器 → time_block_template_sets + time_block_templates
                    ↓ 应用到日期范围
              time_blocks（date-based 实例）
                    ↓ 关联
              tasks.time_block_id → 日历展示
                    ↓ Agent
              get_time_blocks(from, to) → gap detection → time_block_setup proposal
```

v1.7.3 完成后：
- 用户编辑模板 → 模板只是蓝图
- 应用模板 → 生成独立实例
- 改某天 → 只影响那天
- Agent 排学习计划 → 查具体日期的实例，有就直接用，没有就提议创建
