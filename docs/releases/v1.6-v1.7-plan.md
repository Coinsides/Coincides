# Coincides v1.6 / v1.7 — 版本规划

> **主题**：知识图谱整合 — Course 中心化 + Card-Task 联动 + UI 结构重组
> **前置版本**：v1.5.3（Agent 智能规划升级）
> **目标**：将 Course 从简单列表升级为知识中心页，打通 Task ↔ Card 双向关联，合并冗余导航项，建立 Course → Goal → Task → Card 的完整知识图谱链路。

---

## 一、设计宪法守则

| 宪法条款 | v1.6/v1.7 约束 |
|---|---|
| 不替用户做决定 | Card-Task 关联由用户或 Agent proposal 提出，用户审批；Section 强制有但名称由用户/Agent 决定 |
| 不监控用户 | 不追踪卡片复习情况来自动调整任务；不根据 checklist 完成率生成未请求的分析 |
| 不制造挫败感 | Task 查看模式不显示完成率统计；未关联 Card 的任务不标红/不警告 |

---

## 二、需求总览

### 需求来源

用户第 6 条消息（含两张截图：Deck 列表页 + Deck 详情页），提出以下核心诉求：

1. **Course 详情页**：Course 成为 hub，展示下属 Goal、Card Deck、统计信息
2. **侧边栏 Review 合并**：Review 从独立导航项移除，改为 Deck 列表页的操作按钮
3. **Card Section 强制化**：禁止无 Section 的卡片，UI + Agent 双重约束
4. **Agent 按章节创建 Section**：批量创建卡片时必须先按章节创建 Section
5. **Task-Card M:N 关联**：Checklist 条目 ↔ Card 映射，任务查看模式显示关联卡片
6. **Task 查看/编辑模式分离**：点击任务先进入只读查看，再点编辑进入编辑模式
7. **Task 查看模式的 Card 气泡**：查看模式下展示关联的卡片气泡（可点击展开）

### 版本分配

| 版本 | 主题 | 核心内容 |
|------|------|----------|
| **v1.6** | Course 中心化 + UI 结构重组 | Course 详情页、侧边栏精简（Review 合并）、Section 强制化、Agent Section 协议 |
| **v1.7** | Task-Card 联动 | Task-Card 关联表、Task 查看/编辑双模式、Card 气泡显示、Checklist→Card 映射 |

**分版理由**：v1.6 是 UI 结构层面的重组（不涉及新数据表），v1.7 是数据模型扩展（新增关联表 + 全新交互模式）。两者解耦，v1.6 可独立交付。

### 4.3 Card Section 强制化

**目标**：所有卡片必须属于某个 Section，禁止"无分类"卡片。

**现状问题**：
- DB schema: `section_id TEXT REFERENCES card_sections(id) ON DELETE SET NULL` → 允许 NULL
- Agent 工具: `section_id` 描述为 "Omit for unsectioned"（definitions.ts line 169）
- 但 Agent prompt 已经说 "Every card MUST have a section_id"（矛盾）

**变更清单**：

| 文件 | 变更 |
|------|------|
| `server/src/db/migrations/` | 新迁移：(1) 为所有 section_id=NULL 的卡片创建 "未分类" Section 并赋值；(2) 不改 DDL（SQLite 不支持 ALTER COLUMN NOT NULL），改在应用层强制 |
| `server/src/routes/cards.ts` | `POST /api/cards` + `PUT /api/cards/:id` — 校验 `section_id` 非空，否则 400 |
| `server/src/agent/tools/definitions.ts` | `create_card` 工具：`section_id` 改为 required，描述改为 "必须提供" |
| `server/src/agent/tools/executor.ts` | `create_card` handler：如果未传 section_id 则报错（不再默认 null） |
| `client/src/components/CardModal/CardModal.tsx` | 创建卡片时 section 选择器必选，不允许留空 |
| `client/src/pages/Decks/DeckDetail.tsx` | "未分类" Section 仍然显示（但标记为系统 Section，不可删除） |

### 4.4 Agent 按章节创建 Section 协议

**目标**：Agent 批量创建卡片时，必须先按文档章节创建 Section，再将卡片分配到对应 Section。

**现状**：
- Agent prompt 已说 "Every card MUST have a section_id"
- Agent 有 `create_card_section` 工具（待确认）
- 但没有强制要求"先创建 Section 再创建 Card"的协议

**变更清单**：

| 文件 | 变更 |
|------|------|
| `server/src/agent/system-prompt.ts` | 新增「Section-First Card Creation 协议」：批量创建卡片前必须 (1) 分析文档章节结构 (2) 为每个章节创建 Section (3) 再批量创建卡片挂到对应 Section |
| `server/src/agent/tools/definitions.ts` | 确认 `create_card_section` 工具存在且参数正确；如不存在则新增 |
| `server/src/agent/tools/executor.ts` | 对应 handler |

**Agent 协议示例**：
```
## Section-First Card Creation（章节优先卡片创建）

当批量创建卡片时（通常来自文档），你必须：
1. 分析文档结构，提取章节/小节标题
2. 调用 create_card_section 为每个章节创建 Section（name = 章节标题）
3. 将卡片分配到对应的 section_id
4. 绝不创建无 section_id 的卡片

如果文档没有明确章节结构，创建一个以文档名为 name 的 Section 作为兜底。
```

## 五、v1.7 详细设计

### 5.1 Task-Card 关联数据模型

**目标**：建立 Task ↔ Card 的多对多关系，支持 Checklist 条目级别的精细映射。

**新增数据表**：

```sql
-- Task-Card 多对多关联
CREATE TABLE IF NOT EXISTS task_cards (
  id TEXT PRIMARY KEY,
  task_id TEXT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  card_id TEXT NOT NULL REFERENCES cards(id) ON DELETE CASCADE,
  checklist_index INTEGER,  -- 对应 task.checklist JSON 数组的 index（NULL 表示任务级关联）
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  UNIQUE(task_id, card_id, checklist_index)
);

CREATE INDEX IF NOT EXISTS idx_task_cards_task ON task_cards(task_id);
CREATE INDEX IF NOT EXISTS idx_task_cards_card ON task_cards(card_id);
```

**关联粒度说明**：
- **任务级关联**：`checklist_index = NULL`，表示整个 Task 关联此 Card
- **Checklist 条目级关联**：`checklist_index = 0/1/2/...`，表示某个 checklist item 对应此 Card

**API 变更**：

| 端点 | 方法 | 描述 |
|------|------|------|
| `/api/tasks/:taskId/cards` | GET | 获取任务关联的所有 Card（含 checklist_index） |
| `/api/tasks/:taskId/cards` | POST | 新增 Task-Card 关联 `{ card_id, checklist_index? }` |
| `/api/tasks/:taskId/cards/:cardId` | DELETE | 删除关联 |
| `/api/cards/:cardId/tasks` | GET | 获取 Card 关联的所有 Task（反向查询） |

**Agent 工具扩展**：

| 工具 | 变更 |
|------|------|
| `create_task` | 新增可选参数 `card_ids?: string[]`，创建任务时同时建立关联 |
| `link_task_cards` | **新工具**：批量建立 Task-Card 关联 `{ task_id, links: [{ card_id, checklist_index? }] }` |

**目标**：Agent 在创建学习计划时，能自动建立 Task ↔ Card 的关联。

**System Prompt 新增**：
```
## Task-Card Linkage（任务-卡片关联）

当创建包含 checklist 的学习任务时：
1. 如果相关 Deck 中已有对应知识点的 Card，使用 link_task_cards 建立关联
```

## 九、与长期架构的关系

本次 v1.6/v1.7 是知识图谱化的第一步：

```
Course ──┬── Goal ── Task ──┬── Checklist Item
         │                  └── Card[] (via task_cards)
         ├── Card Deck ── Section ── Card
         └── Document
```

后续可扩展方向：
- Card 反向追溯：从 Card 查看关联的所有 Task（"这张卡片是哪些任务要求学的"）
- Goal 完成度：基于下属 Task 完成率 + Card 复习状态综合计算
- Document → Card 溯源：卡片已有 `source_document_id`，可在 Course 详情页展示文档 → 卡片的生成关系
