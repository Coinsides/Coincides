# v2.0 Experience Review - NoteBlock Foundation

**Created**: 2026-05-19
**Branch**: `feat/v2.0-noteblock`
**Experience Status**: `PASS_WITH_FOLLOWUP`

This file is the product experience state machine for v2.0. It records UI, interaction, workflow, visual density, accessibility, and product-fit findings. Engineering quality findings belong in `docs/releases/v2.0-review.md`.

---

## Current Experience Status

`PASS_WITH_FOLLOWUP`

Henry accepted the current NoteBlock experience as sufficient for v2.0 Foundation. The experience is functional but visibly early. The issues below are not blockers for Step 1, but they should guide v2.1/v2.2 or a dedicated NoteBlock product pass.

---

## UI / Visual Findings

### Finding: Note editor feels like an engineering test surface

Severity: Medium
Type: Visual / Product Fit
Release decision: `DEFER_ACCEPTED`

Current behavior:

- A newly created note opens as `Untitled note`.
- The first visible surface is a plain title field plus a simple add-block form.
- The page communicates storage mechanics more than a finished learning-note experience.

Expected product-grade behavior:

- New notes should feel intentionally created, with a calmer first state and better default affordances.
- The page should suggest a learning workflow, not only a database editor.

Suggested version:

- v2.1 or v2.2 NoteBlock UX polish.

---

## Interaction Findings

### Finding: NoteBlock reorder uses up/down buttons

Severity: Medium
Type: Interaction
Release decision: `DEFER_ACCEPTED`

Current behavior:

- Reordering blocks works through `Move up` and `Move down` buttons.
- This proves the ordering model and API, but it feels primitive compared with product-grade editors.

Expected product-grade behavior:

- Drag-and-drop reorder, keyboard-accessible reorder, or a more spatial interaction model.
- Clearer placement feedback when moving blocks.

Suggested version:

- v2.1 or v2.2, depending on whether NoteBlock UX polish becomes the next product priority.

### Finding: Block type list is fixed and math-heavy

Severity: Medium
Type: Interaction / Product Fit
Release decision: `DEFER_ACCEPTED`

Current behavior:

- Available types include `paragraph`, `definition`, `theorem`, `proof`, `formula`, `example`, `exercise`, `answer`, and `sidenote`.
- Users cannot create a custom type such as `concept`.

Expected product-grade behavior:

- User-definable or course/package-defined block types.
- User language should not be limited to the initial math-oriented set.

Suggested version:

- v2.2 or later, unless custom block types become required for v2.1 AI Note Proposal.

---

## Workflow Findings

### Finding: Add Note starts from a bare untitled state

Severity: Low
Type: Workflow / Copy
Release decision: `DEFER_ACCEPTED`

Current behavior:

- `Add Note` immediately creates an `Untitled note`.
- The user must infer what to do next from the editor controls.

Expected product-grade behavior:

- Better default naming, inline first-run guidance, or a lightweight note creation prompt.
- The user should feel they are starting a learning artifact, not filling a blank database record.

Suggested version:

- v2.1/v2.2 UX polish.

---

## Product Fit Findings

### Finding: Foundation is structurally right but not yet a learning companion experience

Severity: Medium
Type: Product Fit
Release decision: `DEFER_ACCEPTED`

Current behavior:

- v2.0 establishes the durable NoteBlock substrate.
- The product experience remains closer to a manual block editor than a learning steward.

Expected product-grade behavior:

- Future versions should connect NoteBlocks to source material, AI proposals, review projections, and more natural study workflows.

Suggested version:

- v2.1+ product direction, aligned with the roadmap and future AI Note Proposal work.

---

## Accessibility / Density Findings

### Finding: NoteBlocks are visually large and only stack vertically

Severity: Medium
Type: Density / Layout
Release decision: `DEFER_ACCEPTED`

Current behavior:

- Blocks occupy large vertical space.
- Layout is linear; blocks cannot arrange side-by-side or adapt to content size.

Expected product-grade behavior:

- Denser editing modes for repeated study work.
- Future support for more flexible layouts where the learning material benefits from it.

Suggested version:

- v2.2 or a dedicated NoteBlock layout pass.

---

## Deferred UX Debt

- Improve empty/new note first impression.
- Add better default title behavior or creation prompt.
- Support custom or package-defined block types.
- Replace or supplement up/down reorder with drag-and-drop or richer keyboard interactions.
- Reduce block visual bulk and support denser editing.
- Explore flexible layouts beyond one-column stacking.

---

- `ACCEPTED`: v2.0 Step 1 foundation is acceptable as a foundation release.
- `DEFER_ACCEPTED`: current NoteBlock UX limitations do not block Step 1.
- `RELEASE_APPROVED`: v2.0 is released as the NoteBlock Foundation baseline.
- Pending for v2.1 planning: decide whether NoteBlock UX polish belongs in v2.1, v2.2, or a separate product pass after AI Note Proposal planning.
