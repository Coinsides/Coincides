# Coincides v1.8 — 云端部署 + PWA 离线支持

> **主题**：从本地单机工具转型为在线服务，支持多用户注册和离线使用  
> **前置版本**：v1.7.3（功能完整的学习系统）  
> **里程碑**：v1.x → v2.0 的桥梁——用户可以注册账号、在线使用、离线复习  
> **方向变更**：原 v1.8 计划为 Electron 桌面打包，经评估后改为云端部署（详见 ADR-0007）

---

## 一、为什么放弃 Electron

Electron 打包过程中暴露了四层兼容性问题：

1. **jiti v2 无 CJS 入口** — Electron 的 CommonJS 环境无法加载 jiti/register
2. **Electron Node ABI ≠ 系统 Node ABI** — native 模块（better-sqlite3, sqlite-vec）无法跨运行时加载
3. **jiti ESM hooks 的 Windows bug** — resolve() 和 load() 返回裸路径而非 file:// URL
4. **架构不匹配** — 运行时 TS 编译 + ESM + native 模块的组合与 Electron 天然冲突

更关键的是：v2.0 的社区功能（模板市场、协作）天然需要云端，Electron 只能作为离线壳。与其先做壳再做云端，不如直接做云端 + PWA 离线，一步到位。

---

## 二、目标架构

```
用户浏览器（或 PWA 安装到桌面）
    ↕ HTTPS
Cloudflare CDN（静态资源 + SSL + 全球加速）
    ↕
Railway / Fly.io（Express 服务端）
    ↕
PostgreSQL（Supabase / Neon 托管数据库）

离线时：
  Service Worker 拦截请求 → IndexedDB 本地缓存 → 恢复网络后同步
```

## 三、实施步骤

### Step 1：清理 Electron 残留 + 仓库整理

**目标**：干净的代码库，移除所有 Electron 相关依赖和配置

- 删除 `electron/` 目录（main.ts, tsconfig.json, server-loader.cjs）
- 删除 `scripts/patch-jiti-windows.cjs`
- `package.json` 移除 electron/electron-builder 依赖和相关 script
- 删除 `dist-electron/`、`release/` 输出目录
- `.gitignore` 清理 Electron 相关条目
- 保留 `server/src/index.ts` 中的 express.static（云端部署同样需要）
- 保留 `DB_PATH` / `UPLOAD_DIR` 环境变量支持（云端同样需要）

### Step 2：SQLite → PostgreSQL 迁移

**目标**：数据库从本地文件切换到云端托管 PostgreSQL

- 安装 `pg`（node-postgres）替代 `better-sqlite3`
- 创建 PostgreSQL schema（从现有 schema.sql 转换）
  - `INTEGER` → `INTEGER`（兼容）
  - `TEXT` 保持不变
  - `REAL` → `DOUBLE PRECISION`
  - `BOOLEAN` → 原 SQLite 用 0/1，PostgreSQL 用原生 `BOOLEAN`
  - `DATETIME` → `TIMESTAMPTZ`
  - 移除 sqlite-vec 向量类型（后续用 pgvector 替代）
- 改写 `server/src/db/init.ts`：
  - 连接 PostgreSQL（`DATABASE_URL` 环境变量）
  - Migration 机制改用 PostgreSQL DDL
- 改写所有 SQL 查询：
  - `?` 占位符 → `$1, $2, $3`（pg 的参数化查询）
  - `db.prepare().run()` → `pool.query()`
  - `db.prepare().get()` → `pool.query()` + `rows[0]`
  - `db.prepare().all()` → `pool.query()` + `rows`
  - 事务语法调整（`BEGIN` / `COMMIT` / `ROLLBACK`）
- 向量搜索暂时降级为 FTS（全文搜索），后续用 pgvector 恢复

### Step 3：多用户支持强化

**目标**：从单用户本地应用升级为多用户在线服务

- 数据库所有表添加 `user_id` 字段（外键关联 users 表）
- 所有查询增加 `WHERE user_id = $current_user` 过滤
- 注册/登录 API 保持不变（已有 JWT 机制）
- 文件上传改用云端存储（S3 / Cloudflare R2）替代本地 `uploads/` 目录
- API Key 管理：每个用户独立的 Anthropic API Key（存储在数据库中，加密）

### Step 4：部署上线

**目标**：应用可以通过 URL 访问

- 客户端 `vite build` → 静态文件部署到 Cloudflare Pages
- 服务端部署到 Railway（或 Fly.io）
  - 环境变量：`DATABASE_URL`, `JWT_SECRET`, `ANTHROPIC_API_KEY`（默认 key，用户可覆盖）
  - 部署钩子：git push → 自动构建部署
- 数据库使用 Supabase 或 Neon（托管 PostgreSQL，免费层足够初期使用）
- 域名 + Cloudflare DNS + HTTPS
- CORS 配置调整（从 `*` 改为具体域名）

### Step 5：PWA 离线支持

**目标**：用户可以"安装"到桌面，断网时仍能复习卡片

- `client/public/manifest.json` — 应用名称、图标、主题色
- `client/src/service-worker.ts` — 缓存策略：
  - 静态资源（JS/CSS/HTML）：Cache First
  - API 数据：Network First + IndexedDB fallback
- 离线数据层：
  - 用户数据（卡片、任务、课程）同步到 IndexedDB
  - 离线操作记录到 pending queue
  - 恢复网络后批量同步
- 离线可用功能：
  - ✅ 卡片复习（FSRS 算法前端已有）
  - ✅ 查看任务/课程/目标
  - ✅ 编辑卡片内容
  - ❌ Agent 对话（需要 API 调用）
  - ❌ 文档上传/解析
- Vite PWA 插件（`vite-plugin-pwa`）简化配置

### Step 6：数据迁移工具（可选）

**目标**：现有本地 SQLite 用户可以把数据导入云端

- 导出脚本：读取本地 SQLite → 生成 JSON
- 导入 API：上传 JSON → 写入 PostgreSQL
- 一次性工具，不需要长期维护

## 四、安全策略

| 层级 | 措施 |
|------|------|
| 传输 | HTTPS（Cloudflare 自动证书），全程 TLS 加密 |
| 认证 | JWT + httpOnly cookie + 短期过期 + refresh token |
| 密码 | bcrypt 哈希（已有） |
| API Key | AES-256 加密存储，不明文保存在数据库 |
| 数据隔离 | 所有查询强制 `user_id` 过滤，防止越权访问 |
| 注入防护 | 参数化查询（pg 原生支持），不拼接 SQL |
| 数据库 | 自动每日备份 + 时间点恢复（PITR） |

## 七、对 v2.0 的铺垫

本版本完成后，v2.0 所需的基础设施全部就位：

- ✅ 多用户系统 → 社区功能的前提
- ✅ PostgreSQL → 支持复杂查询、关系型数据、pgvector
- ✅ 云端部署 → 模板市场、用户协作的基础
- ✅ 文件云存储 → 社区共享文档/模板
- ✅ PWA → 移动端覆盖，不需要额外开发 native app
